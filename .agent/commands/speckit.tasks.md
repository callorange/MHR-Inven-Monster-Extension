## 사용자 입력 (User Input)

```text
$ARGUMENTS
```

진행하기 전에 사용자 입력값이 있다면 **반드시** 이를 고려해야 합니다.

## 사전 실행 확인 (Pre-Execution Checks)

**확장 훅(Extension Hooks) 확인 (태스크 생성 이전)**:
- 프로젝트 루트에 `.specify/extensions.yml` 파일이 존재하는지 확인합니다.
- 존재한다면 파일을 읽어 `hooks.before_tasks` 키 아래의 항목들을 찾습니다.
- YAML 파싱이 실패하거나 유효하지 않은 경우, 훅 확인 과정을 조용히 건너뛰고 정상적으로 계속 진행합니다.
- `enabled`가 명시적으로 `false`인 훅은 제외합니다. `enabled` 필드가 없는 훅은 기본적으로 활성화된 것으로 간주합니다.
- 남은 훅들에 대해 `condition` 표현식을 직접 해석하거나 평가하려고 시도하지 **마십시오**:
  - 훅에 `condition` 필드가 없거나 비어 있는 경우, 해당 훅을 실행 가능한 것으로 간주합니다.
  - 훅에 내용이 있는 `condition`이 정의된 경우, 해당 훅을 건너뛰고 조건 평가는 HookExecutor 구현체에 맡깁니다.
- 실행 가능한 각 훅에 대해 `optional` 플래그에 따라 다음 중 하나를 출력합니다:
  - **선택적 훅** (`optional: true`):
    ```
    ## 확장 훅 (Extension Hooks)

    **선택적 사전 훅(Optional Pre-Hook)**: {extension}
    명령어: `/{command}`
    설명: {description}

    프롬프트: {prompt}
    실행하려면: `/{command}`
    ```
  - **필수 훅** (`optional: false`):
    ```
    ## 확장 훅 (Extension Hooks)

    **자동 사전 훅(Automatic Pre-Hook)**: {extension}
    실행 중: `/{command}`
    EXECUTE_COMMAND: {command}
    
    Outline 단계로 넘어가기 전에 이 훅 명령어의 수행 결과를 기다리십시오.
    ```
- 등록된 훅이 없거나 `.specify/extensions.yml`이 존재하지 않으면 조용히 건너뜁니다.

## 개요 (Outline)

1. **설정(Setup)**: 저장소 루트에서 `.specify/scripts/powershell/check-prerequisites.ps1 -Json` 명령어를 실행하고 FEATURE_DIR 및 AVAILABLE_DOCS 목록을 파싱합니다. 모든 경로는 반드시 절대 경로(absolute path)여야 합니다. "I'm Groot"와 같이 문자열 내 작은따옴표가 포함된 경우 이스케이프 구문(e.g., 'I'\''m Groot')을 사용하거나 가능하면 큰따옴표(e.g., "I'm Groot")를 사용합니다.

2. **설계 문서 로드(Load design documents)**: FEATURE_DIR에서 문서를 읽습니다:
   - **필수**: plan.md (기술 스택, 라이브러리, 구조), spec.md (우선순위가 지정된 사용자 스토리)
   - **선택**: data-model.md (엔티티), contracts/ (인터페이스 컨트랙트), research.md (의사결정 사항), quickstart.md (테스트 시나리오)
   - 참고: 모든 프로젝트가 위 문서를 다 갖고 있지는 않습니다. 사용 가능한 문서들을 바탕으로 태스크를 생성하세요.

3. **태스크 생성 워크플로우 실행(Execute task generation workflow)**:
   - plan.md를 불러와서 기술 스택, 라이브러리, 프로젝트 구조 추출
   - spec.md를 불러와서 사용자 스토리 및 우선순위(P1, P2, P3 등) 추출
   - data-model.md가 존재할 경우: 엔티티를 추출하여 관련된 사용자 스토리에 매핑
   - contracts/ 가 존재할 경우: 인터페이스 컨트랙트를 관련된 사용자 스토리에 매핑
   - research.md가 존재할 경우: 초기 설정(setup) 작업에 필요한 의사결정 내역 추출
   - (아래의 '태스크 생성 규칙'에 따라) 사용자 스토리별로 그룹화된 태스크 생성
   - 사용자 스토리 진행 순서를 보여주는 의존성(dependency) 그래프 생성
   - 각 사용자 스토리당 병렬 실행 예제 작성
   - 태스크 무결성( completeness) 검증 (각 스토리가 독립적 테스트 가능토록 모든 태스크를 온전히 포함하는지)

4. **tasks.md 생성**: `.specify/templates/tasks-template.md`를 템플릿 구조로 사용하며, 다음 내용으로 채웁니다:
   - plan.md에 기재된 정확한 기능 이름
   - 1단계(Phase 1): 초기 설정 태스크 (프로젝트 프로젝트 초기화)
   - 2단계(Phase 2): 기반 태스크 (모든 사용자 스토리 전 선제 완료되어야 하는 블로킹 사전 조건)
   - 3단계(Phase 3+): 각 사용자 스토리 당 1개의 페이즈(단계)부여 (spec.md의 우선순위 순서대로)
   - 각 단계에는 다음이 포함되어야 합니다: 스토리의 핵심 목표, 독립적인 테스트 기준, 요구될 경우 테스트 작성, 구현 태스크 목록
   - 마지막 단계(Final Phase): 마무리 수정 작업(Polish) 및 공통 관심 영역 처리
   - 모든 태스크는 반드시 엄격한 체크리스트 형식을 따라야 합니다 (아래 '태스크 생성 규칙' 참고)
   - 각 태스크에 대해 명확한 파일 경로 표기
   - 사용자 스토리 완료 순서가 명시된 의존성(Dependencies) 섹션 표기
   - 스토리 별 병렬 실행 가능한 작업 예시
   - 구현 전략(Implementation strategy) 섹션 (MVP 우선적용, 증분 배포 전략)

5. **보고(Report)**: 생성된 tasks.md의 경로 및 요약 정보 출력:
   - 총 태스크 수
   - 각 사용자 스토리별 태스크 수
   - 식별된 병렬 작업 기회
   - 각 스토리에 대한 독립적 테스트 기준 명시
   - 제안되는 MVP 스코프 (보통 사용자 스토리 1만 해당)
   - 양식 검증: 모든 태스크가 체크리스트 형식(체크박스, ID, 라벨, 수정할 파일 경로)을 따르는지 확인 진행

6. **확장 훅(Extension Hooks) 확인 (완료 이후)**:
   - 이후의 동작은 before_tasks와 유사하게 `hooks.after_tasks`를 확인하여 훅을 실행/보고합니다.
   
명령어 적용 컨텍스트: $ARGUMENTS

**생성되는 tasks.md는 즉시 실행 가능한 형태여야 합니다**: 각 태스크의 내용만 읽어도 추가적인 정보 없이 LLM이 충분히 코드를 짤 수 있도록 아주 구체적으로 작성되어야 합니다.

## 태스크 생성 규칙 (Task Generation Rules)

**주의 (CRITICAL)**: 각 사용자 스토리는 독립적으로 구현 및 단위 테스트가 가능하도록 반드시 사용자 스토리 단위로 병렬 그룹화되어야 합니다.

**테스트 코드는 선택 사항(OPTIONAL)입니다**: 기능 명세서에 명시적으로 요구되었거나 사용자가 TDD 방식을 요청한 경우에만 테스트 작성 태스크를 생성하십시오.

### 체크리스트 양식 (REQUIRED)

반드시 모든 태스크가 다음 양식을 엄격하게 준수해야 합니다:

```text
- [ ] [TaskID] [P?] [Story?] 명확한 설명 및 파일 경로
```

**양식 구성요소**:

1. **체크박스(Checkbox)**: 반드시 `- [ ] ` (마크다운 체크박스 형식)부터 시작합니다.
2. **Task ID**: 순차적 고유 번호 부여 (실행 순서대로 T001, T002, T003...)
3. **[P] 마커**: 다른 파일들을 동시에 수정하거나 불완전한 태스크들에 서로 의존성이 없어 '병렬 처리'가 가능한 작업일 경우에만 표시합니다.
4. **[Story] 라벨 지시자**: '사용자 스토리 단계(User Story phase)'에 위치한 태스크에 대해서만 '필수(REQUIRED)' 입니다.
   - 형식: [US1], [US2], [US3] 등 (spec.md의 사용자 스토리와 대응됨)
   - 설정(Setup) 단계: 스토리 라벨 표기 안 함
   - 기반(Foundational) 단계: 스토리 라벨 표기 안 함
   - 사용자 스토리(User Story) 단계: 스토리에 해당하는 라벨 필수 표기
   - 마무리/다듬기(Polish) 단계: 스토리 라벨 표기 안 함
5. **설명(Description)**: 명확한 액션 및 정확한 파일의 절대명(절대경로) 표기

**예시**:

- ✅ 올바름: `- [ ] T001 구현 계획(plan.md)에 따른 프로젝트 구조 생성`
- ✅ 올바름: `- [ ] T005 [P] src/middleware/auth.py 경로에 인증(authentication) 미들웨어 구현`
- ✅ 올바름: `- [ ] T012 [P] [US1] src/models/user.py 경로에 User 모델 생성`
- ✅ 올바름: `- [ ] T014 [US1] src/services/user_service.py 경로에 UserService 로직 구현`
- ❌ 틀림: `- [ ] User 모델 생성` (ID와 스토리 라벨이 누락됨)
- ❌ 틀림: `T001 [US1] 모델 생성` (체크박스가 누락됨)
- ❌ 틀림: `- [ ] [US1] User 모델 생성` (Task ID가 누락됨)
- ❌ 틀림: `- [ ] T001 [US1] 모델 생성` (파일 대상 경로가 누락됨)

### 태스크 분류 규칙 (Task Organization)

1. **사용자 스토리 (spec.md 기준)** - 최우선 분류 기준:
   - 각각의 사용자 스토리 (P1, P2, P3...)는 하나의 개별적인 단계(phase)로서 블록화됩니다.
   - 모든 관련 컴포넌트를 현재 속한 스토리의 블록 내 매핑시킵니다:
     - 스토리가 요구하는 모델(Models)
     - 스토리가 요구하는 서비스(Services)
     - 스토리가 요구하는 인터페이스/UI
     - 테스트 작성이 필요한 경우: 해당 스토리를 위한 지정 테스트 코드
   - 스토리들 간의 의존성(Dependencies)을 표기합니다 (대부분의 사용자 스토리는 타 스토리로부터 독립적이어야 합니다).

2. **인터페이스 (Contracts 기준)**:
   - 각 인터페이스 컨트랙트를 → 해당 인터페이스를 제공하는 스토리에 맵핑
   - 테스트 작성이 필요한 경우: 인터페이스 건별로 [P] 태그가 달린 컨트랙트(contract) 테스트 태스크를 소속 스토리 구현 전단에 명시

3. **데이터 모델 (Data Model 기준)**:
   - 각 엔티티 설계를 요구하는 특정 사용자 스토리에 맞게 엔티티 태스크를 매핑
   - 엔티티 하나가 여러 스토리에 걸쳐서 다중 적용될 경우: 가장 먼저 발생하는 스토리 단계 혹은 기초 기반(Setup) 단계에 배치
   - 엔티티 간 연결 관계 맺음 설계 → 해당 서비스 계층 구현 태스크로 적절한 스토리에 편입시킴

4. **설정/인프라스트럭쳐 (Setup/Infrastructure 기준)**:
   - 공유되는 공통 인프라 아키텍처 → Phase 1. 기본 설정 단계 (Setup phase)
   - 다른 태스크에 앞서 선행 필수 불가결한 태스크 → Phase 2. 기반 태스크 단계 (Foundational phase)
   - 특정 유닛 스토리에 한해서 적용되는 모듈 설정 → 해당 유닛 스토리 단계 내부에 편입

### 실행 단계 구조 (Phase Structure)

- **Phase 1**: 기반 설정 (프로젝트 초기 세팅)
- **Phase 2**: 선행 기초 단계 (블로킹 요소 - 반드시 사용자 스토리 시작에 앞서서 완료되어야만 함)
- **Phase 3+**: 스펙(spec)의 우선순위 내림차순(P1, P2, P3...)으로 한 덩어리씩의 사용자 스토리가 이어짐
  - 하나의 유닛 스토리 안에서는: 테스트 코드 (선택적) → 모델 계층 → 서비스 계층 → 엔드포인트 계층 → 통합(Integration) 로직 순으로 개발
  - 각 단계는 온전히 작동하고 독립적인 테스트가 가능한 한계 수준(increment)이어야 함
- **기타 최종 Phase**: 마무리 및 다듬기, 포괄적인(Cross-Cutting) 관심사 해결 관련 작업 그룹표기
