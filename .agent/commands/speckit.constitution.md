## 사용자 입력 (User Input)

```text
$ARGUMENTS
```

진행하기 전에 사용자 입력값이 있다면 **반드시** 이를 고려해야 합니다.

## 개요 (Outline)

당신은 `.specify/memory/constitution.md` 파일에 존재하는 프로젝트 헌법(constitution)을 업데이트하는 임무를 맡고 있습니다. 이 파일은 대괄호 안에 플레이스홀더 토큰(예: `[PROJECT_NAME]`, `[PRINCIPLE_1_NAME]`)을 포함하는 형태의 TEMPLATE(템플릿) 파일입니다. 
당신의 작업 절차는 다음과 같습니다: (a) 구체적인 값을 수집/도출해내고, (b) 템플릿의 빈칸을 정확하게 채워 넣으며, (c) 수정 사항이 발생했다면 종속된 다른 아티팩트(문서)에도 이 변경 사항을 전파하는 것입니다.

**참고**: 프로젝트 설정(setup) 단계에서 `.specify/templates/constitution-template.md`를 기반으로 `.specify/memory/constitution.md`가 초기화되어 생성되어야 맞습니다. 만약 아직 해당 위치에 헌법 파일이 없다면 먼저 템플릿을 복사해 오십시오.

다음 실행 흐름(execution flow)을 따르십시오:

1. `.specify/memory/constitution.md`에 위치한 기존 헌법 문서를 불러옵니다.
   - `[ALL_CAPS_IDENTIFIER]`(대문자 식별자) 형식의 모든 플레이스홀더 토큰을 찾아냅니다.
   **중요**: 사용자는 템플릿에 명시된 원칙의 개수보다 더 적거나 혹은 더 많은 항목을 요구할 수도 있습니다. 특정 개수를 원한다면 일반적인 템플릿 포맷을 따르되 그 개수를 존중하여 문서를 맞게 업데이트하십시오.

2. 플레이스홀더를 채울 구체적 값 도출 및 수집:
   - 사용자 입력(대화형 컨텍스트 등)으로 얻은 값이 있다면 그 값을 사용합니다.
   - 그렇지 않으면 기존의 저장소 환경(README, 문서, 이전 버전의 헌법 등)에서 문맥을 추론해 이끌어냅니다.
   - 거버넌스 날짜 표기: `RATIFICATION_DATE`(비준일)은 헌법이 최초 제정된 날짜입니다 (모를 경우 물어보거나 임시로 TODO 마커를 답니다). `LAST_AMENDED_DATE`(최종 개정일)는 변경 사항이 일어나는 오늘 날짜이며 내용 변경이 없었다면 이전 날짜를 그대로 유지합니다.
   - `CONSTITUTION_VERSION` (헌법 버전)은 반드시 시맨틱 버저닝(Semantic versioning) 규칙에 따라 숫자를 올려야 합니다:
     - MAJOR(메이저): 하위 호환성이 깨어지는 수준의 거버넌스/원칙 삭제 처리나 재정의가 일어난 경우.
     - MINOR(마이너): 새로운 원칙이나 섹션이 하나 추가되거나, 지침 내용이 물질적으로 눈에 띄게 확장된 경우.
     - PATCH(패치): 모호함 해소, 단어/오타 수정, 비의미론적인 구조 정제.
   - 버전 상승(bump) 단계를 정하기 애매한 경우, 최종 확정 짓기 전에 변경 사유를 먼저 제안하십시오.

3. 업데이트된 헌법 콘텐츠 초안 작성:
   - 모든 플레이스홀더를 구체적인 실제 텍스트로 치환합니다 (고의로 프로젝트 내에서 아직 결정하지 않기로 합의하여 의도적으로 남겨둔 상황이 아니라면 어떠한 빈칸 괄호 토큰도 남겨두어선 안 됩니다—만약 남긴다면 명시적으로 이유를 정당화해야 합니다).
   - 기존의 헤딩(heading) 계층 구조를 보존하십시오. 안내용 주석은 토큰을 성공적으로 치환 완료했다면 안내 구실을 잃었으므로 지워도 무방합니다. (단, 삭제 시 의미가 모호해질 우려가 있는 주석은 유지합니다).
   - 각각의 원칙(Principle) 섹션에 다음이 포함되도록 확인하십시오: 간결한 원칙 제목 1줄, 결코 타협할 수 없는 규칙을 명시한 설명 문단(혹은 리스트), 그리고 명백하지 않을 땐 그 원칙이 도입된 구체적 이유(rationale).
   - 거버넌스(Governance) 섹션에 '어떻게 개정하는가(amendment procedure)', '버전 상승 정책(versioning policy)', '컴플라이언스 준수 리뷰 기대치(compliance review expectations)'가 잘 명세되어 있는지 점검하십시오.

4. 일관성 전파 및 검증 체크리스트 (이전 양식을 활성 유효성 검사로 변환 활용):
   - `.specify/templates/plan-template.md`를 읽고 헌법 점검(Constitution Check) 혹은 규칙 조항들이 새로 업데이트된 원칙들과 잘 정렬되어 맞물리는지 확인합니다.
   - `.specify/templates/spec-template.md`를 읽고 스코프/요구사항 한계를 판별합니다—만약 헌법상에서 필수 섹션 혹은 제약조건의 추가나 삭제가 있었다면 템플릿도 이에 맞게 수정합니다.
   - `.specify/templates/tasks-template.md`를 읽고 태스크 생성 시 분류 카테고리 항목들이 새로 생기거나 지워진 헌법 원칙(예: 관측 가능성 중시, 버전관리 필수, 테스트 규율 등)을 제대로 반영해주고 있는지 확인합니다.
   - `.specify/templates/commands/*.md` 디렉토리 하부 파일들(이 파일도 포함)을 열람하여, 제너릭한 공용 규칙이 적용되어야 할 곳에 구식의 레거시 용어(가령 CLAUDE처럼 특정 에이전트 편향명)가 남아있지 않은지 검토합니다.
   - 런타임 환경 설정 지침 문서 등(`README.md`, `docs/quickstart.md` 혹은 특정 에이전트 전용 가이드 파일 등)을 살펴봅니다. 원칙이 변경되었다면 참조된 문서들도 찾아 들어가 업데이트하십시오.

5. 동기화 영향 보고서(Sync Impact Report) 생성 (헌법 업데이트 직후 파일 최상단에 HTML 코멘트 양식으로 덧붙여 주입합니다):
   - 변경된 버전 정보: 과거 버전 → 신규 버전
   - 수정된 원칙들의 목록 (타이틀이 바뀌었다면 구 타이틀 → 새 타이틀 병기)
   - 새롭게 추가된 섹션 목록
   - 삭제된 섹션 목록
   - 업데이트 일괄 적용이 요구되는 종속 템플릿 파일들의 경로 (처리 상태 표기: ✅ updated / ⚠ pending)
   - 일부러 플레이스홀더를 미뤄두었다면 진행해야 할 후행 TODO 항목 리스트.

6. 최종 결과물 산출 전 유효성 검사 (Validation):
   - 이유 없이 해소되지 않고 덩그러니 남은 괄호 토큰이 일절 없어야 합니다.
   - 파일 내 기재된 버전 표기 문구와 리포트상의 버전이 일치해야 합니다.
   - 날짜 포맷은 무조건 ISO 규격인 YYYY-MM-DD 형식을 지켜야 합니다.
   - 원칙은 단호한 선언형이어야 하며, 테스트/검증이 가능해야 하고, 모호한 표현의 사용을 피해야 합니다 (예: 막연히 "should (~해야 한다)" → 구체적인 원인/합리성이 포함된 "MUST/SHOULD" 표기로 교체 검토).

7. 완성된 헌법을 `.specify/memory/constitution.md` 파일에 디스크 저장(덮어쓰기)합니다.

8. 수정된 내용에 관한 최종 요약본을 화면에 출력하여 사용자에게 보고합니다:
   - 새 버전 넘버링 및 버전 상승을 일으킨 구체적 사유.
   - 자동 치환에 실패하여 사용자가 수작업으로 변경/체계화를 챙겨야 한다고 판단되어 경고 깃발(flag)이 붙은 파일 내역.
   - 제안하는 Git 커밋 메시지 (예: `docs: amend constitution to vX.Y.Z (principle additions + governance update)`).

포맷팅 및 스타일 필수 요구사항:

- 템플릿과 완전히 똑같은 뎁스의 마크다운 헤딩(`##` 등) 단계를 그대로 이용하십시오 (임의로 레벨을 강등시키거나 승격시키지 마세요).
- 가독성을 위해 내용이 긴 이유(rationale) 텍스트의 경우 가로줄 바꿈을 넣어 주십시오 (가급적 100자 미만); 하지만 일부러 단어를 보기 흉하게 끊어 먹는 식의 강제 줄 바꿈은 지양하십시오.
- 각 섹션 사이에 반드시 하나의 빈 줄을 둡니다.
- 문장 맨 끝에 보이지 않는 공백(trailing whitespace)이 남지 않게 주의하십시오.

가령 사용자가 (원칙 한 개만 수정해달라 하는 것처럼) 아주 부분적인 업데이트만 요청했을 때에도, 이 유효성 검사와 버전을 어떻게 올릴지에 대한 결정 단계는 모두 수행해야만 합니다.

만약 주요 정보가 빠져있다면 (예: 최초 제정 날짜를 도무지 추론할 수 없을 때), 빈칸에 `TODO(<해당_필드명>): 사유설명` 포맷으로 끼워 넣고, 이 이슈를 '동기화 영향 보고서(Sync Impact Report)' 내 보류된 항목으로 리포팅 하십시오.

절대로 새로운 엉뚱한 템플릿 파일을 새로 만들지 마십시오; 항상 기존의 `.specify/memory/constitution.md` 파일을 베이스로 덮어쓰며 작업하십시오.
