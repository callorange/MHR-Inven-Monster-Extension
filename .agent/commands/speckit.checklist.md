## 체크리스트의 목적: "영문서(요구사항)에 대한 유닛 테스트 (Unit Tests for English)"

**핵심 개념 (CRITICAL CONCEPT)**: 보통의 테스팅이 코드 작동을 점검하듯, 이 체크리스트는 **'요구사항 작성 품질에 대한 유닛 테스트(UNIT TESTS FOR REQUIREMENTS WRITING)'** 입니다. 특정 도메인 내 쓰여진 요구사항의 내용적 완성도(quality), 명료성(clarity), 그리고 무결성(completeness)을 검증하기 위한 것입니다.

**검증/기능 테스트용이 아닙니다 (NOT for verification/testing)**:

- ❌ 틀림: "버튼 클릭이 올바르게 작동하는지 확인하기"
- ❌ 틀림: "에러 핸들링이 잘 돌아가는지 테스트하기"
- ❌ 틀림: "API가 200 응답 코드를 주는지 확인하기"
- ❌ 틀림: 구현된 코드가 명세서와 일치하는지를 점검하는 것

**요구사항 품질 검증용입니다 (FOR requirements quality validation)**:

- ✅ 맞음: "모든 카드 유형에 대한 시각적 계층화 요구사항이 빠짐없이 정의되었는가?" (완성도)
- ✅ 맞음: "'눈에 띄는 위치에 표시'라는 요구조건이 특정한 픽셀 크기/위치 값으로 수치화되었는가?" (명료성)
- ✅ 맞음: "마우스 호버(hover) 상태에 대한 요구사항이 모든 반응형 컴포넌트에 일관성 있게 규정되었는가?" (일관성)
- ✅ 맞음: "키보드 위주 탐색(Keyboard navigation)을 진행할 때의 접근성(a11y) 기준이 명확히 정의되었는가?" (커버리지/분석 반경)
- ✅ 맞음: "로고를 불러오지 못했을 때 어떻게 동작해야 하는지 명세하고 있는가?" (예외 상황 처리/엣지 케이스)

**메타포 (Metaphor)**: '개발 명세서(spec)가 곧 자연어(한국어 혹은 영어)로 짜인 코드'라고 가정한다면, 이 체크리스트는 그것을 돌려보는 테스트 스위트(Test suite)입니다. 요구사항이 잘 쓰였는지, 명확한지, 다 빠짐없이 기재되었는지, 나중에 구현할 수 있을 정도로 잘 설계되었는지를 시험하는 것입니다. 기능이 잘 동작할 것인지를 테스트하는 것이 절대 아님을 명심하십시오.

## 사용자 입력 (User Input)

```text
$ARGUMENTS
```

진행하기 전에 사용자 입력값이 있다면 **반드시** 이를 고려해야 합니다.

## 실행 단계 (Execution Steps)

1. **설정(Setup)**: 저장소 루트에서 `.specify/scripts/powershell/check-prerequisites.ps1 -Json` 명령어를 실행하고 JSON 결과를 파싱하여 `FEATURE_DIR` 경로와 `AVAILABLE_DOCS` 목록을 읽어옵니다.
   - 모든 경로는 절대 경로여야 합니다.
   - "I'm Groot"와 같은 작은따옴표를 전달할 땐 이스케이프 구문(e.g., 'I'\''m Groot') 또는 가능하면 큰따옴표(e.g., "I'm Groot")를 사용하세요.

2. **의도 명확화 - 동적 문답 (Clarify intent - dynamic)**: 꽉 막힌 카탈로그가 아닌 지능적인 추론을 통해 초기 맥락을 잡아내기 위한 질문을 최대 세 가지(3개)로 도출하십시오. 이 질문들은 반드시:
   - 사용자가 입력한 구문과 spec/plan/tasks 파일에서 얻을 수 있는 정보에서 생성되어야 합니다.
   - 체크리스트 내용물에 가시적이고 물질적인 변화를 이끌어 낼 수 있는 질문이어야만 합니다.
   - 이미 `$ARGUMENTS`나 단서들로 모호함이 없다면 그 질문은 묻지 않고 건너뛰어야 합니다.
   - 질문의 포괄성보단 디테일한 질문 해상도(정확성)를 선호하십시오.

   생성(Generation) 알고리즘:
   1. 정보 단서 추출: 기능 영역 키워드 (예: 인증, 대기시간(latency), UX, API), 위험성 지표 ("중요한(critical)", "반드시(must)", "컴플라이언스"), 이해당사자 힌트 ("QA", "리뷰", "보안팀"), 그리고 명시된 인도물 규격 ("a11y", "롤백 처리", "컨트랙트/API 명세").
   2. 집중해야 할 영역(candidate focus areas)을 4개 이하로 그룹화한 뒤 우선순위를 매깁니다.
   3. 체크리스트를 읽어볼 수용 대상(작성자, 리뷰어, QA 담당, 배포 담당)과 타이밍이 구체적으로 나오지 않았다면 이를 식별합니다.
   4. 누락된 설계 차원 감지: 커버 스코프 너비(breadth), 깊이/엄격함(rigor), 리스크 위주(risk emphasis), 범위 외 배제 조건(exclusion boundaries), 정량적(측정 가능한) 인수 기준.
   5. 다음 원형(archetypes)들 중 적합한 것으로 질문을 구성하십시오:
      - 스코프 한정용 (예: "로컬 모듈에서만 완벽히 돌게 할까요, 아니면 X, Y 시스템 등과의 연계 지점까지 이 검사 목록에 담을까요?")
      - 리스크 타기팅용 (예: "아래 영역들 중 필수 방어벽(방어 조건)으로 어떤 위험 요소를 중점 체크하도록 할까요?")
      - 심도 조율용 (예: "그냥 푸시 전 자가 점검용으로 한두 줄 짜드릴까요, 아니면 엄격한 릴리즈 승인용으로 세심하게 짜드릴까요?")
      - 독자 타기팅 (예: "수정할 본인만 쓸 리스트입니까, 아니면 PR 리뷰어들이 지적할 때 참고할 용도입니까?")
      - 예외 범위 한정 (예: "성능 문제 관련해서는 이번 주기 체크리스트에서 명시적으로 제외 처리할까요?")
      - 시나리오 갭 탐색 (예: "에러 대처에 대한 명세서가 잘 보이지 않는데요 — 롤백 절차나 부분적인 실패 처리 요건들도 고려 대상입니까?")

   질문 양식 규칙:
   - 옵션을 제공할 경우 작고 콤팩트한 테이블 형태를 사용하십시오 (예: 옵션 | 후보 항목 | 이것이 중요한 이유).
   - 선택지는 최대 A~E 옵션 안으로 끝내십시오; 주관식이 더 깔끔하다고 판단되면 표를 생략해도 됩니다.
   - 사용자가 이미 말했던 것을 다시 앵무새처럼 반복하게 만드는 질문은 절대로 묻지 마십시오.
   - 잊힌 범주를 추론 위주로 날조하지(hallucinate) 마십시오. 확실치 않다면 "X 부분도 포함 범위에 들어가는 게 맞는지 확인 바랍니다." 하고 명확히 물어보십시오.

   질문 불가능/무응답 상황에 대비한 기본값(Defaults):
   - 심도 설정(Depth): 표준(Standard)
   - 타기팅 대상(Audience): 코드 결함 쪽이면 PR 리뷰어 기준; 그것이 아니면 명세서 작성자 본인 기준
   - 집중 영역(Focus): 가장 연관성 높은 2개의 그룹 영역

   질문들을 화면에 출력하십시오 (Q1/Q2/Q3 라벨 부여 요망). 답변을 들은 후 분석을 해봤는데도 '대안/예외/장애 복구/비기능 요구사항' 등 최소 2개 이상의 메이저 시나리오 정보가 확보되지 못했다면, 왜 물어보는지를 한 줄로 요약한 추가 심화 질문(targeted follow-ups)을 최대 두 개(Q4/Q5)까지만 더 늘려서 묻습니다(예: "아직 해결되지 않은 복구 프로세스의 리스크가 높다고 판단되어 하나 더 여쭤봅니다"). 어떠한 경우에도 도합 5번째 질문 스크롤을 넘겨서는 안 되며, 사용자가 더 이상의 질문을 거부할 경우 대기 없이 바로 진행하십시오.

3. **사용자 요청 사항 이해**: `$ARGUMENTS` + 추가로 거친 Q&A 내용을 통합하십시오:
   - 체크리스트의 주요 테마를 도출 (테마 예: 보안, 코드 리뷰, 배포 점검, 사용자 경험(UX))
   - 사용자가 특별히 챙겨달라 했던 '꼭 들어가야 할 항목(must-have)'들이 누락되지 않도록 정리
   - 사용자가 뽑아준 집중 포인트(focus selections)를 카테고리 뼈대에 맵핑
   - spec/plan/tasks 등 기존 산출물들을 읽으면서 누락된 맥락을 유추 추가 (절대로 없는 기능을 소설처럼 날조하지 마세요)

4. **기능 컨텍스트 불러오기 (Load feature context)**: `FEATURE_DIR`에서 이하의 정보 문서들을 읽어옵니다:
   - spec.md: 기능 요건 및 개발 스코프 범위
   - plan.md (존재하는 경우): 기술적 디테일, 종속성 스택 관리
   - tasks.md (존재하는 경우): 구현체 태스크 리스트업

   **컨텍스트 로딩 전략 (Context Loading Strategy)**:
   - 지정받은 포커싱 영역에 한해서만 가장 필요하고 유관한 부분만 읽어 들이십시오 (단순 텍스트 복붙 나열식 로딩 절대 금지).
   - 길게 늘어진 본문 텍스트보다는 요약된 시나리오/불릿(bullet) 요구사항 위주의 숏폼 정보를 환영하십시오.
   - 점진적인 정보 공개 원칙을 지키십시오 (초기 로딩 이후, 결락된 부분만 한정 타겟팅하여 추가 쿼리 호출 실시).
   - 참조하는 원본 문서 크기가 심각하게 비대할 경우, 그대로 임베딩하지 말고 요약 항목들을 생성해서 축약하십시오.

5. **체크리스트 생성 (Generate checklist)** - "요구사항 검증용 단위 테스트(Unit Tests for Requirements)" 파일 생성:
   - `FEATURE_DIR/checklists/` 디렉토리가 없다면 생성하십시오.
   - 고유한 이름을 가진 체크리스트 파일을 만드십시오:
     - 도메인 특성에 맞춘 짧고 명료한 이름 (예: `ux.md`, `api.md`, `security.md`)
     - 파일 이름 포맷: `[domain].md`
   - 파일 저장/수정 방식:
     - 해당 이름의 파일이 존재하지 않는 경우: 새로운 파일을 생성하고, 항목 ID는 CHK001부터 부여하십시오.
     - 해당 이름의 파일이 이미 존재하는 경우: 기존 파일 내용에 새 항목들을 추가(Appended)하십시오. ID 숫자는 마지막 번호부터 이어받습니다 (예를 들어 마지막 항목이 CHK015면, 신규 항목은 CHK016으로 시작).
     - 어떠한 경우에도 기존 체크리스트의 요건 항목들을 마음대로 지우거나 초기화(Replace)해 버리면 안 됩니다. 변경 없이 내용 보존 후 밑단에 덧붙여(Append) 주십시오.

   **핵심 원칙 (CORE PRINCIPLE) - 구현물이 아닌 요구사항 명세 자체를 테스트하라**:
   반드시 각각의 체크리스트 문항 항목들은 요구사항 명세 문서(REQUIREMENTS THEMSELVES)가 아래 기준을 만족하는지 검사해야 합니다:
   - **완성도 (Completeness)**: 빠진 내용 없이 반드시 필수 요구사항이 다 적혔는가?
   - **명료성 (Clarity)**: 요구조건을 읽었을 때 다의적이거나 애매하지 않고 누구나 알기 쉽게 적혔는가?
   - **일관성 (Consistency)**: 요구사항들끼리 서로 모순되는 요구를 하고 있진 않은가?
   - **측정/테스트 가능성 (Measurability)**: 이 요구사항을 만족했는지 아닌지를 제 3자가 수치를 재서 파악할 수 있는가?
   - **커버리지 (Coverage)**: 일어날 수 있는 다양한 시나리오나 최악의 상황/모서리 사례(edge case)가 모두 망라되었는가?

   **카테고리 구성 방식** - 요구조건 품질 판단 기준 별 그룹핑 (Category Structure):
   - **Requirement Completeness** (필수 요구사항이 명세서에 전부 문서화되어 있는가?)
   - **Requirement Clarity** (요구사항 명세가 명확하고 뚜렷하게 정의되어 있는가?)
   - **Requirement Consistency** (요구사항 간 충돌 없이 일관되게 정렬되어 있는가?)
   - **Acceptance Criteria Quality** (성공 기준 지표가 수치로 측정 가능한 형태인가?)
   - **Scenario Coverage** (가능한 모든 주요 기능/장애 시나리오들이 다 고려되었는가?)
   - **Edge Case Coverage** (극한 환경에서의 경계 조건 처리가 잘 명세되었는가?)
   - **Non-Functional Requirements** (성능, 보안, 접근성 지침 등도 빠짐없이 명시되었는가?)
   - **Dependencies & Assumptions** (의존 스택과 전제 상황들이 철저히 문서화되고 검증되었는가?)
   - **Ambiguities & Conflicts** (해석의 차이가 생기기 전에 명확화(Clarification)해야 할 부분이 있는가?)

   **체크리스트 문항 작성 가이드 (HOW TO WRITE CHECKLIST ITEMS)** - 이른바 "영어(명세서)에 대한 유닛 테스트 작성법":

   ❌ **틀린 예시 (WRONG)** (구현 결과를 테스트함):
   - "랜딩 페이지가 3개의 에피소드 카드를 표시하는지 검증한다."
   - "데스크톱에서 마우스 호버 상태가 제대로 작동하는지 테스트한다."
   - "로고를 클릭했을 때 홈 화면으로 넘어가는지 확인한다."

   ✅ **올바른 예시 (CORRECT)** (요구사항 문서의 질을 테스트함):
   - "추천 에피소드 노출 개수와 위치 레이아웃이 정확한 개수와 위치로 지정되어 적혀 있는가?" [Completeness/완성도 파트]
   - "'눈에 띄는 표시'라는 말이 주관적이지 않도록 구체적인 객체 해상도 수치나 위치 치수로 변환되었는가?" [Clarity/명료성 파트]
   - "모든 대화형 요소를 관통하여 시각적 호버 상태 반응 규칙이 문서 내에서 일관되게 적용되었는가?" [Consistency/일관성 파트]
   - "탭(Tab) 키보드로만 이동해야 하는 사용자를 위한 인터랙션 지침이 누락되지 않고 적혀 있는가?" [Coverage/커버리지 파트]
   - "로고 이미지 렌더링에 실패했을 때 나타날 폴백 설계(Fallback behavior) 동작이 명세되어 있는가?" [Edge Cases/예외 파트]
   - "비동기 에피소드 데이터를 가져오는 동안 표시할 로딩 애니메이션 상태 요구 구간이 설계되었는가?" [Completeness/완성도 파트]
   - "어떤 UI 요소가 다른 요소보다 우선적으로 눈에 띄게 할 지에 관한 시각적 계층화(visual hierarchy) 구조가 명의되어 있는가?" [Clarity/명료성 파트]

   **문항 구조 (ITEM STRUCTURE)**:
   각 문항은 다음 패턴을 반드시 따릅니다:
   - 요구조건의 질(quality)에 대해서 물어보는 형식을 취해야 합니다 (이런 내용 있습니까? 저런 내용 구체적으로 적혀 있나요?)
   - 철저히 작성된 "기록(spec/plan)"에만 초점을 두어야 합니다 (혹은 적히지 않고 빠진 내용)
   - 속성값을 괄호 안에 표기합니다: [Completeness / Clarity / Consistency / 기타 등등...]
   - 요구 사항을 이미 검사한 경우 문서 내 섹션 번호 등 참조를 병기합니다: `[Spec §X.Y]`
   - 빠진 요건을 검사하고 있을 경우 구멍 난 곳을 마킹합니다: `[Gap]`

   **품질 측정 지표별 문항 예시들 (EXAMPLES BY QUALITY DIMENSION)**:

   완성도 (Completeness):
   - "발생 가능한 모든 API 실패 모드(400, 401, 500 등)별 예외 처리 규정이 정의되어 있는가? [Gap]"
   - "모든 인터랙티브 액션들에 대한 접근성 기준이 설계되어 있는가? [Completeness]"
   - "반응형 디자인을 위한 각 디바이스 뷰포트별 모바일 레이아웃 브레이크포인트 단위가 세밀히 적혀 있는가? [Gap]"

   명료성 (Clarity):
   - "'빠른 로딩 시간' 이라는 추상적 문구가 최소 로딩 기준 시간(Threshold timing) 수치 기준으로 명시화되었는가? [Clarity, Spec §NFR-2]"
   - "'연관 에피소드 추천'을 판단하는 검색 로직 및 필터 조건(선택 기준)이 정확하게 정의되었는가? [Clarity, Spec §FR-5]"
   - "'적당히 눈에 띄게(Prominent)'라는 요구사항이 검증할 수 있는 시각 속성(색, 두께 등) 기준 단위로 표현되었는가? [Ambiguity, Spec §FR-4]"

   일관성 (Consistency):
   - "전체 네비게이션 동작 요구사항이 모든 독립된 페이지 디자인에 걸쳐 똑같이 일관되게 준수되는가? [Consistency, Spec §FR-10]"
   - "상세 페이지에 위치한 카드 컴포넌트 정보도 랜딩 페이지 디자인과 엇박자 나지 않게 일관된 스펙인가? [Consistency]"

   커버리지 (Coverage):
   - "데이터가 아직 아무것도 입력되지 않은 제로 상태 (Zero-state / No episodes) 상황에 대한 화면 스펙 디자인이 기획되었는가? [Coverage, Edge Case]"
   - "동일 데이터에 동시다발적으로 접근할 때 처리를 위한(Concurrent user interaction) 시나리오도 취급하는가? [Coverage, Gap]"
   - "데이터의 부분적 로딩 실패 처리에 대한 스펙 시나리오가 마련되어 있는가? [Coverage, Exception Flow]"

   측정/테스트 가능성 (Measurability):
   - "시각 계층화 구조 요구사항을 테스트 코드 수준으로 측정 가능하거나 검증 가능하게 짤 수 있는가? [Acceptance Criteria, Spec §FR-1]"
   - "'시각적으로 균형 잡힌 가중치(Balanced visual weight)' 라는 요건을 테스터가 주관적이지 않게 객관적으로 판별할 수 있는가? [Measurability, Spec §FR-2]"

   **시나리오 식별 및 대응 관점 평가 (Scenario Classification & Coverage)**:
   - 다음 관점의 시나리오가 모두 담겨있는지 평가합니다: 주요 뷰(Primary), 대체 뷰(Alternate), 예외/에러(Exception/Error), 장애 복구 뷰(Recovery), 그리고 비기능 디자인 시나리오(Non-Functional).
   - 각 시나리오에 대해 물어봅니다: "명세된 [어느 시나리오 유형] 에 대한 요건이 누수 없이 일관되고 명확한가?"
   - 시나리오 자체가 누락된 경우: "[해당 시나리오 유형] 파트를 의도적으로 뺐는가 아니면 기재를 깜빡했는가? [Gap]"
   - 서버 상태값 변형이 수반되는 시스템일 경우: "마이그레이션 변경 조치가 실패했을 상황을 염두에 둔 롤백 대안 매뉴얼이 있는가? [Gap]"

   **추적성 및 이력 남기기 (Traceability Requirements)**:
   - 최소 조건: 전체 생성된 문항 중 최소 80% 이상의 문항이 어디서 나온 출처인지 레퍼런스가 달려 있어야 합니다.
   - 각 아이템은 다음과 같은 참조 출처를 괄호에 표기해야 합니다: `[Spec §X.Y]` 또는 `[Gap]`, `[Ambiguity]`, `[Conflict]`, `[Assumption]` 등을 마커로 표시.
   - 프로젝트에서 ID를 부여하는 포맷 자체가 없다면: "개별 요구사항 및 인수 테스트 케이스들에 대한 추적용 ID(scheme scheme) 체계가 잡혀있는가? [Traceability]" 를 체크 항목에 두십시오.

   **요구조건 관련 제반 품질 이슈 확인 (Surface & Resolve Issues)**:
   요구조건 문서 자신에 대해 이렇게 꼬투리 잡아 탐색하십시오:
   - 애매함: "설마 'fast' 따위의 말을 측정 기준도 마련 없이 무책임하게 지껄여놓진 않았는가? [Ambiguity, Spec §NFR-1]"
   - 상충 발생: "여기선 왼쪽이라고 해놓고 [§FR-10a] 저기선 오른쪽이라고 우기고 있진 않은지? [Conflict]"
   - 검증 안 된 전제: "'서드파티 API 서버는 결코 죽지 않고 항상 가용하다'는 위험한 망상을 가정하고 스펙을 짰는가? [Assumption]"
   - 종속성 누락: "그 팟캐스트 API 서버 연동에 대한 연결 요구조건이나 스펙 명세를 찾을 수 없는 게 정상인가? [Dependency, Gap]"
   - 개념 및 사전 미정의: "대체 '시각적 계층화 요소'가 무엇을 의미하고 어떤 기준으로 검증한다는 건가요? [Gap]"

   **결괏값 콘텐츠의 취합/융합 전략 (Content Consolidation)**:
   - 소프트 캡 리미트: 무작위 생성 항목이 40개가 넘어가면 그중 리스크가 크거나 파급력이 높은 항목들만 살려내십시오.
   - 동일 항목을 체크하지만 수식어만 다른 말장난 같은 중복성 문구들은 과감히 쳐내 통합하십시오.
   - 파급력이 극도로 떨어지는 너무 지엽적인 엣지 케이스만 5개가 넘을 경우 아예 쿨하게 하나로 뭉쳐 버리십시오: "X, Y, Z 같은 엣지 케이스들을 스펙 요구사항으로 다 감아냈음? [Coverage]"

   **🚫 이 행동 절대 금지 (ABSOLUTELY PROHIBITED)** - 이렇게 쓰면 요구조건 명세서가 아니라 완성된 소스 코드 프로그램 보고 시비 거는 테스트 문구입니다:
   - ❌ "검증(Verify)", "테스트(Test)", "확인(Confirm)", "체크(Check)" 류의 행동 강령 단어와 구현물의 행위를 결합하는 표현 절대로 금지 (명세서 내용의 검증일 경우 제외. 완성된 앱의 기능 검증일 경우 불가)
   - ❌ 코드 실행 관련 기능, 사용자의 GUI 이용 행위, 혹은 시스템 작동 기능 여부 테스트 등 언급 불가
   - ❌ "올바르게 표시됨", "기능이 온전히 돌아감", "원하던 대로 작동함" 과도 같은 표현
   - ❌ "클릭해라", "안쪽으로 항해해라(navigate)", "렌더링해라", "로딩해라", "실행해 봐라"
   - ❌ 일반적인 자동화 테스트 도구가 진행하는 테스트 케이스 검증 멘트, 테스트 플랜 서식, 혹은 QA 담당자의 점검 절차 멘트 포함
   - ❌ "리액트로 짰음?", "API 알고리즘 구현함?" 따위의 소셜 엔지니어링 틱한 상세 구현 코딩 기법 질문 포함 불가

   **✅ 따라야 할 구조적 예시 문장 패턴 (REQUIRED PATTERNS)** - 요구조건 설계(Requirements Quality) 자체를 묻는 질문:
   - ✅ "그 상황[scenario]에 대비하여 이렇게 대응하라고 스펙/문서/요구조건[requirement type] 상에 명시되어(documented) 있습니까?"
   - ✅ "그 [애매모호한 용어/vague term] 표현이 명확한 수치나 구체적인 스펙 기준으로 재정의 및 문서화되어 있습니까?"
   - ✅ "[문단 A]에서 요구하는 스펙과 [문단 B]에서 요구하는 스펙이 서로 모순 없이 설계되었습니까?"
   - ✅ "[요구된 기능 스펙]이 과연 (사람의 주관이 섞이지 않고) 프로그램이나 산술적으로 객관적인 측정이 가능하게 문서화되었습니까?"
   - ✅ "해당 [기능/엣지 케이스]가 요구조건 기재 사항에 포함되어 있긴 한 겁니까?"
   - ✅ "기획서나 명세서가 그 [누락된 스펙/missing aspect]에 대한 작동 방향 구조를 제대로 꼼꼼히 짰나요?"

6. **구조 참조 모델 작성 기준 (Structure Reference)**: `.specify/templates/checklist-template.md`의 표준 템플릿 양식을 바탕으로 제목(Title), 설정 메타 데이터 영역, 카테고리 헤더 타이틀 등을 잡고 ID 규칙에 맞춰 리스트를 생성합니다. 템플릿 이용이 불가능한 오류 환경이라면: 일단 H1 급 Title로 시작, Meta 데이터 작성 후, 최소 2개의 샾기호 `##` 카테고리들 내부에 순차적으로 증분 기입되는 글로벌 넘버링 ID (`- [ ] CHK### <요구조건검증문>`)들을 CHK001 부터 순차 배치하십시오.

7. **보고 (Report)**: 최종 생성된 checklist 파일의 저장 절대 경로 안내, 만들어 넣은 검사 문항(items)의 개수, 완전 새 파일을 창조했는지 아니면 기존 파일 끝단에 항목을 이어서(appended) 편입시켰는지에 대한 로그를 표기하십시오. 또한 작성하면서 활용한 다음과 같은 파라미터 컨텍스트의 요약 정보를 추가로 표기하십시오:
   - 집중적으로 파고들었던 포커싱 선정 영역 (Focus areas selected)
   - 분석 점검 깊이 수준 (Depth level)
   - 주요 수검 대상 타겟/시점 (Actor/timing)
   - 사용자 측이 무조건 넣어 달라고 했던 필수 아이템 요소 내역들

**중요 사항**: 각각의 `/speckit.checklist` 명령어 호출 시 도메인 이름을 가진 알아보기 쉽고 짧은 템플릿 전용 이름을 가지게 되며 항상 새 파일을 만들거나 원래 파일 뒷전에 이어서 항목을 추가합니다. 이렇게 함으로써 다음과 같은 효과를 만듭니다:

- 각 도메인 특성에 집중한 개별의 점검 목차 작성 (`ux.md`, `test.md`, `security.md` 등)
- 체크리스트의 성격을 가리키는 간단하고 직관적인 기억되기 쉬운 이름 지어주기
- `checklists/` 폴더 안에서의 용이한 항목 식별 검색 보조 기능 사용 지원

나중에 쌓여서 지저분해지는 걸 고려해 항상 타입 명세를 잘 적어주시고, 불필요해진 항목이 보이면 쓸모없는 리스트는 삭제 청소를 지원해 주십시오.

샘플 문항:

- "발생 가능한 모든 시스템 오류, 서버 뻑남 현상 등에 관한 예외 응답 전문 양식이 미리 약속되어 적혀 있는가? [Completeness]"
- "Rate Limiting의 트래픽 차단 요구 사항이 두리뭉실하지 않고 특정한 호출 제한 수치 임계치 단위(Thresholds)로 세밀하게 제시되어 있는가? [Clarity]"
- "개발하는 모든 엔드포인트들의 인증, 권한 확인 조건 설계 스펙이 다른 모듈들과도 상호 충돌 없이 일관되게 규격화되었는가? [Consistency]"
- "외부 플러그인 네트워크 및 통신 지연에 대응할 외부 의존성 관련 실패 상황 재시도(Retry)/타임아웃(Timeout) 룰 정책이 존재하고 문서화되어 있는가? [Coverage, Gap]"
- "요구 스펙 문서에 시스템의 릴리스 버전 관리 체계(Versioning Strategy)에 관한 조항이 누락되진 않았는가? [Gap]"

**성능 요건 점검 (Performance Requirements Quality) 샘플 파일:** `performance.md`

샘플 문항:

- "요구하는 성능 지표가 무의미한 미사여구(ex. 번개처럼 빠름)가 아니라 구체적인 성능 지표 단위로 정의되어 있는가? [Clarity]"
- "사용자의 핵심 이용 동선 전반에 걸친 세부 조작 구간별 성능 목표 지수 수치가 전부 확립되어 있는가? [Coverage]"
- "트래픽이 폭주하는 극한의 부하 환경 조건별 성능 저하 타겟 허용 범위에 관한 대응책 시나리오 모델 설정과 명세가 확립되었는가? [Completeness]"
- "요구하는 성능 수치들을 인간의 개입 없이 완전히 프로그램 코드에 의한 계측 수단 측정으로 증명할 수 있는가? [Measurability]"
- "서버 과부하 처리 셧다운 발생 등 고사양 로드 시발생 시나리오 시 작동을 느려지게만 해서라도 시스템을 유지할 최소 가동 데그라데이션(Degradation) 규칙 룰이 스펙에 존재하는가? [Edge Case, Gap]"

**보안 명세 검증 (Security Requirements Quality) 샘플 파일:** `security.md`

샘플 문항:

- "접근 차단 및 보호 조치가 따르는 모든 정보 리소스에 대해 식별, 가입 및 인증 처리에 대한 필수 가이드 요건이 전부 기재되어 있는가? [Coverage]"
- "개인 정보와 민감한 패스워드 정보를 어떻게 격리 보호할지에 관한 내부 세부 암호화 정책 규칙이 온전히 명세되었는가? [Completeness]"
- "시스템이 당면한 스렛(Threat - 해킹 공격 위협 등) 방호 방어 모델에 관해 언급되어 있고 설계 요구 스펙에 이를 타파할 요건이 들어 있는가? [Traceability]"
- "스펙 설계상 도입한 보안 정책이 법정 국가 및 회사 차원의 정보 보안 준수 법규(Compliance) 조항 조건들과 상충되는 모순 규정이 있지 않고 충실히 일관되는가? [Consistency]"
- "치명적인 보안 위협이 감지되거나 서버 시스템 데이터 탈취 붕괴 시 어떻게 조치해야 할지 사후 장애/침해 사고 롤백 대응 행동 지침 스펙 매뉴얼이 설계에 명기되었는가? [Gap, Exception Flow]"

## 이거 금지 조항 (Anti-Examples: What NOT To Do)

**❌ 금지 (WRONG) - 이건 '명세'가 아니라 '구현 결과'에 시비를 거는 테스트 플랜이잖아:**

```markdown
- [ ] CHK001 - 랜딩 메인 페이지 화면에 멋진 에피소드 카드 이미지가 3장 예쁘게 정가운데 뜨는지 확인 [Spec §FR-001]
- [ ] CHK002 - 데스크탑 화면에서 마우스 오버 시 올바르게 색이 반전하는지 작동 기능 검사 [Spec §FR-003]
- [ ] CHK003 - 브랜드 로고 아이콘을 클릭해서 때리면 루트 홈 페이지로 잘 회귀해서 화면 전환이 정상 처리 되는지 동작 확인 [Spec §FR-010]
- [ ] CHK004 - 릴레이션(유사 콘텐츠) 추천 구간 에피소드 출력 섹션 영역에 적절하게 항목 3~5개가 안 잘리고 깔끔하게 나열되어 표시되는지를 체크 [Spec §FR-005]
```

**✅ 정답 (CORRECT) - 이건 정상적으로 '문서 및 요구사항의 질적 문제'를 검증하는 테스트입니다:**

```markdown
- [ ] CHK001 - 홈화면 랜딩 메인 시 노출시켜야 할 그 메인 '에피소드 피쳐 카드'의 정확한 나열 출력 갯수와 위치가 명시적으로 정해져 있고 틈이 없는가? [Completeness, Spec §FR-001]
- [ ] CHK002 - 해당 마우스 커서 호버 상태에 대한 시각 디자인 변동 사항 정의 문서가 기타 나머지 인터랙티브 컴포넌트들에도 일일이 동일한 룰을 관통해서 보장하는 등 일관적인 룰베이스를 제공하고 있는가? [Consistency, Spec §FR-003]
- [ ] CHK003 - 사용자가 마우스를 눌러 반응할 모든 브랜드 아이콘들에 해당하는 이동 클릭 위치 처리 응답 네비게이션 액션 처리에 관한 규칙이 분명하고 모호함 없이 셋업 및 제공되었는가? [Clarity, Spec §FR-010]
- [ ] CHK004 - 관련 유사 에피소드 데이터 추출용 추천 목록을 긁어오기 위한 항목 선택 분기 알고리즘의 필터링 선정 조건 규약 파라미터가 명확히 확립되어 있는가? [Gap, Spec §FR-005]
- [ ] CHK005 - 비동기로 수초 내 통신 딜레이 시간이 발생할 데이터 통신 때 노출될 임시 대기 시각 이펙트 로딩 상태 디자인 설계가 빠짐없이 등록되었는가? [Gap]
- [ ] CHK006 - 스펙에 명시된 해당 "아주 돋보이는 시각 요소 계층화 처리(visual hierarchy)"라는 말이, 도대체 객관적인 코딩 지표(픽셀 등 제한적 계측 단위)로 측정 가능한 언어로 쓰여있는가? [Measurability, Spec §FR-001]
```

**둘의 극명한 차이점 (Key Differences):**

- 틀림: 완성된 앱 및 프런트 클라이언트 결과의 정상 구동 유무 체크 및 실행
- 올바름: 해당 기능 설계에 적힌 기능 규약 및 설명서에 하자, 비약, 생략이 없는지에 대한 문맥의 오류 가능성에 대해 조사
- 틀림: 클라이언트 시스템 모듈의 기능 및 액션 행위를 확인
- 올바름: 요구조건 질의 우수성을 점토 및 검증
- 틀림: "화면의 버튼이 의도한 기능대로 동작합니까?"
- 올바름: "화면의 버튼 동작 규율이 누구든 명확하게 알 수 있도록 잘 문서에 정의되었습니까?"
