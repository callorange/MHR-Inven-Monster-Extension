## 사용자 입력 (User Input)

```text
$ARGUMENTS
```

진행하기 전에 사용자 입력값이 있다면 **반드시** 이를 고려해야 합니다.

## 개요 (Outline)

목표: 현재 활성화된 기능 명세서(feature specification) 내에서 모호하거나 의사 결정이 누락된 지점을 명확히 파악하고, 명확해진 내용(clarifications)을 spec 파일에 직접 기록합니다.

참고: 이 명확화(clarification) 워크플로우는 `/speckit.plan` 명령어를 실행하기 이전에 실행되어 완료되어야 합니다. 만약 사용자가 명시적으로 명확화 단계를 건너뛰겠다고 선언하는 경우(예: 탐색적 스파이크 테스트 등), 진행할 수는 있으나 추후 재작업(rework) 리스크가 커질 수 있음을 경고해야 합니다.

실행 단계:

1. 저장소 루트에서 `.specify/scripts/powershell/check-prerequisites.ps1 -Json -PathsOnly` (혹은 단축 옵션 병용 `--json --paths-only` / `-Json -PathsOnly`) 명령어를 **단 한 번** 실행합니다. JSON 출력 결과 중 다음의 최소 필수 필드를 파싱합니다:
   - `FEATURE_DIR`
   - `FEATURE_SPEC`
   - (향후 연결 워크플로우를 위해 선택적으로 `IMPL_PLAN`, `TASKS` 도 확보)
   - 파싱에 실패하면 즉시 중단하고 사용자에게 기능 브랜치 환경을 점검하거나 `/speckit.specify`를 다시 실행하라고 안내하십시오.
   - "I'm Groot"와 같이 작은따옴표를 매개변수에 포함해야 할 땐 이스케이프 구문(e.g., 'I'\''m Groot') 또는 가능하면 큰따옴표(e.g., "I'm Groot")를 사용하세요.

2. 현재 spec 파일을 불러옵니다. 아래의 분류 체계(taxonomy)를 사용하여 구조적인 모호성 및 누락 커버리지 스캔을 수행하십시오. 각 카테고리별로 '완벽함(Clear) / 부분적임(Partial) / 누락됨(Missing)' 상태를 맵핑합니다. 스캔 결과로 내부용 커버리지 맵을 생성하되, 질문할 거리가 전혀 없는 상황이 아니라면 사용자의 화면에 원형 그대로 맵을 노출하지 마십시오.

   기능의 범위 및 동작 (Functional Scope & Behavior):
   - 핵심 사용자 목표(goals) 및 성공 판단 기준
   - 명시적으로 '개발 제외 대상(out-of-scope)' 처리된 항목
   - 사용자 역할(Roles) / 페르소나별 권한 구분

   도메인 및 데이터 모델 (Domain & Data Model):
   - 엔티티(Entities), 속성, 릴레이션(관계)
   - 식별 수단(Identity) 및 고윳값(uniqueness) 규칙
   - 데이터 생명주기(Lifecycle) 및 상태(state) 전환
   - 예상되는 데이터 규모 / 트래픽 확장성(scale) 수준

   인터랙션 및 UX 흐름 (Interaction & UX Flow):
   - 가장 중요한 크리티컬 유저 저니(user journeys) / 조작 순서
   - 에러 발생 동작 / 데이터 텅 빔(empty) 화면 / 로딩 상태
   - 접근성(A11Y) 혹은 다국어 번역(localization) 관련 참고 사항

   비기능적 품질 속성 (Non-Functional Quality Attributes):
   - 성능 (레이턴시(지연 시간), 처리량(throughput) 기댓값)
   - 확장성 (수직/수평적 확장, 상한선 리미트)
   - 신뢰성 및 가용성 (업타임 요구, 장애 복구 기대치)
   - 관측 가능성 (로깅, 메트릭 통계, 트레이싱(Tracing) 신호 등)
   - 보안 및 개인정보 무결성 (인증(AuthN)/인가(AuthZ), 데이터 보호, 위협 및 방어 가정 사항)
   - 법적 준법성 / 규제 제약조건 (컴플라이언스가 존재하는 경우)

   외부 시스템 통합 및 의존성 (Integration & External Dependencies):
   - 연동할 외부 서비스/오픈 API 및 해당 모듈이 뻗었을 때의 대처
   - 데이터 임포트(import) / 엑스포트(export) 포맷
   - 통신 프로토콜 / 버전 관리 가정치

   엣지 케이스 및 실패 핸들링 (Edge Cases & Failure Handling):
   - 부정적인 악의적 작동 시나리오
   - 횟수 제한(Rate limiting) / 과부하 스로틀링(throttling) 제어
   - 충돌 해결책 (예: 여러 유저가 같은 글을 동시 편집(concurrent edits)할 경우 등)

   제약 및 기회비용 (Constraints & Tradeoffs):
   - 기술적 한계 및 제약 (언어, 스토리지, 호스팅 제약)
   - 개발을 위해 의도적으로 수락한 트레이드오프 기수용 항목이나 대안 배제 내역

   용어 및 일관성 (Terminology & Consistency):
   - 전역적으로 사용할 규범적 용어 사전(Glossary terms)
   - 헷갈리게 쓰지 말아야 할 동의어 / 역사 속으로 퇴출된 구형 용어들

   구현 완료 선언 신호 (Completion Signals):
   - 인수 테스트 통과 기준(Acceptance criteria)
   - 측정 가능한 수치 단위의 '완료 기준(Definition of Done)' 지표

   기타 및 임시 마커 (Misc / Placeholders):
   - 해결되지 않고 방치된 TODO 마커 / 미결정 지점
   - 수치 정량화 없이 "견고한", "직관적인" 따위의 모호한 형용사들

   확인 결과 '부분적임(Partial)' 또는 '누락됨(Missing)'으로 판별된 각 부문에 대하여 잠재적인 질문 후보를 추가하십시오. 단, 아래의 경우 질문에서 배제합니다:
   - 물어본들 실제 코드 구현이나 검증 전략에 가시적 물질적 차이를 주지 못하는 뻔한 질문일 경우
   - 명세(spec)를 넘어 나중에 수행될 계획(plan) 단계로 미루는 게 훨씬 이치에 맞을 경우 (이 사유는 별도로 내부 기록해 둡니다)

3. 잠재적 질문 후보 큐(가장 물어봐야 할 우선순위)를 내부적으로 최대 5개까지 생성합니다. 5개를 한꺼번에 와르르 던지지 마십시오. 다음 제약 사항을 준수하십시오:
    - 전 세션을 통틀어 **총합 최대 5개의 질문**만 허용됩니다.
    - 각 질문은 둘 중 하나의 형태로만 대답을 얻어내야 합니다:
       - 짧은 객관식 보기 제공 (2~5개의 확연히 구분되고 상호 배타적인 옵션들), 혹은
       - 한 낱말 / 짧은 단어 구 응답 유도 (명시적으로 조건을 겁니다: "5단어 이내로 답변해 주세요").
    - 아키텍처, 데이터 모델, 태스크 세분화, 테스트 디자인, UX의 행동, 라이브 서비스 투입 가능성, 보안/컴플라이언스 준수성에 치명적인 영향을 끼칠 정도로 중요한 질문만을 선별합니다.
    - 카테고리 균형을 맞추십시오: 심각도가 가장 높은 핵심 미해결 영역부터 우선 다루어야 합니다; 예컨대 보안 구성처럼 고위험군 영역이 미해결 상태인데 엉뚱한 버튼 색상 같은 자잘한 걸 두 개 연속 물어봐선 안 됩니다.
    - 이미 답을 알고 있는 질문, 자잘한 코딩 컨벤션 스타일 질문, 그리고 spec 단계가 아니라 plan 단계에서나 결정될 너무 딥(deep)한 질문은 구조적으로 기능 결함을 초래할 수준이 아니라면 배제합니다.
    - 뒷 단계 재작업(rework) 위험을 눈에 띄게 줄여 주거나, 잘못된 인수 테스트 스펙을 미연에 방지할 수 있는 명확화 포인트에 높은 가중치를 둡니다.
    - 5개 이상의 카테고리가 미해결 상태일 경우, 휴리스틱(영향도 * 불확실성 리스크) 값을 따져 최상위 5개를 고릅니다.

4. 순차적 문답 순환 룹 (대화형):
    - 정확히 한 번에 **하나의 질문만** 화면에 출력합니다.
    - 객관식 뷰어(Multiple-choice) 옵션 질문의 경우:
       - **모든 옵션을 선분석** 한 뒤 아래 기준에 의거하여 **가장 이상적인 추천 옵션 1개**를 먼저 결정해 놓으십시오:
          - 프로젝트 유형에 맞는 베스트 프랙티스
          - 유사 시스템 설계 시 흔히 쓰이는 공통 패턴
          - 리스크 경감 (보안, 성능, 유지 보수성)
          - 명세서상에 드러난 프로젝트 고유 목적이나 제약 한계치 준수 여부
       - 사용자에게 제시할 때 **당신의 추천 옵션안을 제일 위쪽에 눈에 띄게 강조**하고 명확한 논리적 근거(왜 이게 베스트인지 1~2문장의 설명)를 함께 남기십시오.
       - 양식(Format): `**추천 (Recommended):** 옵션 [X] - <선택 근거>`
       - 그 아래에 보기 옵션들을 마크다운 기반 표(Table)로 렌더링하십시오:

       | 선택(Option) | 상세 설명(Description) |
       |--------|-------------|
       | A | <A 옵션에 대한 설명> |
       | B | <B 옵션에 대한 설명> |
       | C | <D, E 옵션 등등 필요시 5개까지> |
       | 직접입력 | 주관식으로 짧게 대답(5 단어 이내) (객관식 주관식 혼용 방식이 어울릴 때만 추가) |

       - 표 출력 직후에 안내문을 답니다: `원하시는 알파벳 옵션 기호(예: "A")로 응답해 주시거나, "yes" 혹은 "추천해 준 걸로"라 답하여 제 추천안을 채택해주세요. 아니면 직접 원하시는 짧은 답변을 적어 주셔도 됩니다.`
    - 주관식 단답형 질문의 경우 (미리보기형 옵션을 주기 애매할 때):
       - 베스트 프랙티스에 기반하여 **가장 권장하는 추천 답안** 모범 예시를 하나 먼저 짚어주십시오.
       - 양식: `**제안 (Suggested):** <당신이 제안할 답변 문구> - <왜 그런지 짧은 근거>`
       - 그 아래 안내문: `형식: 짧은 문구 (5단어 이내). 제 제안에 동의하신다면 "yes" 혹은 "제안대로"라 답해주시고, 아니라면 원하시는 답을 직접 입력해 주세요.`
    - 사용자가 어찌 됐건 응답을 마친 후:
       - 사용자가 "yes", "추천안", "제안해 준걸로" 등의 동의의 뜻을 비치면, 미리 말해두었던 권장/제안 옵션을 답변으로 채택합니다.
       - 그게 아니라면 답변이 보기 옵션 표에 존재하는지, 혹은 주관식의 경우 제약 조건(5단어 규약)에 대략 부합하는지 봅니다.
       - 입력값이 애매해서 못 알아들었을 경우, 명확하게 의미를 다시 확인합니다 (이 되묻기 카운트는 횟수로 치지 않고 여전히 같은 번호 질문 단계에 머무릅니다).
       - 사용자의 의도가 성공적으로 파악되면 일단 이를 임시 메모리에 기록해 두고(아직 파일에 쓰지는 않음), 곧바로 큐 내용물을 꺼내 다음 질문으로 넘어갑니다.
    - 아래 상황일 경우 질의 루프를 종료합니다:
       - 덜컥 초반에 중요한 걸 다 물어보고 나서 더 이상 치명적으로 모호한 게 없을 때 (남은 큐 질문 내용들은 하찮거나 불필요한 것으로 격하됨)
       - 사용자가 더 이상의 질의응답을 거부하는 신호를 줄 때 ("끝", "다 됐어", "그만")
       - 질문 횟수 카운트가 최대 상한치 5회에 도달했을 때.
    - 절대 당신의 큐를 까발리거나 앞으로 이런 이런 것들을 물어보겠다는 식의 스포일러를 사전에 늘어놓지 마십시오.
    - 처음부터 물어볼 가치가 있을 만큼 모호한 부분이 단 하나도 발견되지 않았다면 즉시 치명적인 모호함이 없음을 보고하고 자동 종료합니다.

5. 각각의 답변이 승인될 때마다 즉시 발생하는 통합 적용 과정(증분 업데이트):
    - 초기 한 번 로드된 명세서(spec) 내용을 메모리상에 상시 인메모리 업데이트된 최신 상태로 원본 포맷과 유지하십시오.
    - 이번 세션의 첫 번째 답변을 적용할 때:
       - 명세서 내에 `## Clarifications` (명확화) 섹션이 있는지 확인하십시오 (없애면 가이드 템플릿의 Overview 또는 Context 섹션 바로 하단에 우선 생성합니다).
       - 해당 범위 아래 서브 헤더로 그날의 일자인 `### Session YYYY-MM-DD` 표제를 삽입하십시오.
       - 응답이 채택된 즉시 하단에 불릿 포인트 한 줄을 추가합니다: `- Q: <질문내용> → A: <최종답변>`.
    - 그 직후 이 명확화된 결론을 명세서 파일 중 가장 핏이 잘 맞는 섹션에 직접 침투 적용시킵니다:
       - 기능적 모호성 해소 → 'Functional Requirements(기능 요구 사항)' 란을 갱신하거나 항목 새로 추가.
       - 사용자 동작 / 액터 구분 해소 → 'User Stories(사용자 스토리)' 나 'Actors' 항목란의 권한이나 제약 분기 추가 갱신.
       - 데이터 모델 / 엔티티 관련 해소 → 'Data Model(데이터 모델)' 란(필드, 타입, 관계망)에 정렬 훼손 없이 삽입; 새로운 제약 사항 깔끔하게 추가.
       - 비기능적 모호성 해소 → 'Success Criteria > Measurable Outcomes (성공 측정 지표)' 항목 내용 중 "엄청나게 많은 트래픽" 따위의 수치 없는 허풍을 없애고 콕 집은 수치나 명시적 타겟 값으로 갱신.
       - 엣지 케이스 / 실패 처리 흐름 해소 → 'Edge Cases (엣지 케이스) / Error Handling (에러 핸들링)' 항목을 생성(템플릿에 맞춰)하거나 기존 항목 아래 불릿 단위로 추가.
       - 용어 혼선 방지 → 명세서 전 영역의 모든 용어를 하나의 정의로 일체화시킴; 단 오해 방지를 위해 한 번에 한해 `(과거 "X"라고 명명했었음)` 주석 표기 유예.
    - 새 답변으로 인해 구형 텍스트에 남아있던 서술형 모호함이 풀려버린 경우, 구버전 텍스트를 제거하고 새 내용으로 덮어씁니다; 폐기되거나 더 이상 쓸모없는 내용을 남겨둬서 스스로 모순되는 문서를 유지하지 않게 주의하십시오.
    - 컨텍스트 상실 리스크(Context loss)를 줄이기 위해 통합 적용 한 건마다 매번 원본 spec 파일을 디스크에 즉시 저장 갱신하십시오(Atomic 덮어쓰기).
    - 문서 무결성을 보존하십시오: 관련 없는 섹션의 상하 순서를 흔들거나 기존 목차 헤딩 레벨 체계를 마음대로 파괴하지 마십시오.
    - 본문에 살이 통통하게 찐 쓸데없는 나래티브 묘사문을 줄이고 주입되는 내용은 컴팩트하고 테스트 가능한 조건문 위주여야만 합니다.

6. 검증 (각각의 디스크 파일 기록 및 덮어쓰기가 일어난 이후 최종 패스 점검으로 진행):
   - 명확화(Clarifications) 기록 세션 파트에 묻고 들은 답변이 정확히 1:1 불릿포인트 쌍으로 중복 없이 보존 중인지 확인.
   - 요청된 질문 총합이 5개를 절대 안 넘겼는지 점검.
   - 새 답변들에 의해 업데이트된 기존 섹션 및 문단에 여태 모호했던 플레이스홀더 구문들이 잔재하고 있는지 색출 완료.
   - 앞서 적용된 답변으로 인해 오히려 나중에 나온 답변이 모순되는 진술로 충돌을 일으키지 않았는지 체크 (스캔 및 폐기).
   - 마크다운 양식이 유효한지 봅니다. 추가가 승인된 신규 목차는 오직 `## Clarifications`, `### Session YYYY-MM-DD` 둘 뿐이어야 함.
   - 용어 일관성 확보: 업데이트된 전체 문단에 걸쳐 규범적 주용어들이 서로 통일되어 사용 중인지 감지.

7. 변경사항이 모두 갱신된 파일을 최종적으로 `FEATURE_SPEC` 절대 경로로 덮어쓰기 저장 완료합니다.

8. 완료 상황 정리 보고 (질의 끝나거나 도중 조기 중단될 때 모두 수행):
   - 총 몇 번 질문했고 답변받았는지를 알림.
   - 변경되고 다듬어진 결과 spec 파일의 절대 경로 공지.
   - 당신의 손길이 가해져 수정된 문서 내 섹션 이름들(목록).
   - 커버리지 요약 요표(Coverage summary table) 출력 - 당신의 카테고리 분석별 최종 상태(Status 안내): Resolved(모호/누락 파악해서 방금 해결함), Deferred(질문 한계 5회 도달 초과 혹은 이건 나중에 plan 단계에 넘기는 게 맞아서 일단 뒤로 보류함), Clear(원래부터 완벽했음), Outstanding(아직 완전히 모호/누락이 풀리진 않았으나 파급력이 낮아 넘어감).
   - Outstanding 상태나 Deferred 상태가 아직 남아 있다면 지금 이 채로 `/speckit.plan`으로 넘어가도 될지 아니면 plan 다 짜고 난 뒤에 나중에 다시 `/speckit.clarify`를 한 번 더 구동해야 할 지 권장 조언 남기기.
   - 사용자에게 추천되는 다음번 실행 명령어 안내.

행동 규칙 요약 (Behavior rules):

- 파고들 가치가 있는 모호함이 단 하나도 발견되지 않았거나 아주 하찮은 모호성만 있을 경우 질문 루프에 진입하지 마십시오: "더 이상 특별히 짚고 넘어갈 만한 치명적이고 모호한 부분이 없습니다."라고 답하고 다음 명령을 제안하십시오.
- 애초에 요구사항 spec 파일이 디스크에 없을 경우, 사용자에게 `/speckit.specify`를 먼저 선행해라 하고 가이드만 주고 중지할 것 (명령어 멋대로 넘나들면서 spec까지 여기서 만들지 말 것).
- 절대로 묻는 카운트를 5회초과해서 선 넘지 말 것 (사용자가 동문서답해서 재차 되묻는 회수는 카운트에 잡히지 않음).
- 순수 기능 작동의 모호성을 가리기 위함이 아니라면 가급적 기술 스택 이름, 프레임워크 이름 등 개발적 추측성 멘트 등은 자제할 것.
- "멈춰", "다 했어", "이제 넘어가" 등 명시적으로 사용자가 조기 종료를 요구하면 무조건 수용하고 중단하기.
- 문서가 완벽한 나머지 아무런 질의가 필요 없었다면 간단한 커버리지 테이블 요약(전 항목 Clear 기록)만 덩그러니 남겨주고 하산하기.
- 5회 질의 쿼터 제한 때문에 미처 다 못 물어본 고위험(High-impact) 모호성 카테고리 이슈가 남아있을 경우 표의 Deferred 란에 이유와 함께 붉게 깃발 달아서 크게 명시해주기.
