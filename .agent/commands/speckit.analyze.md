## 사용자 입력 (User Input)

```text
$ARGUMENTS
```

진행하기 전에 사용자 입력값이 있다면 **반드시** 이를 고려해야 합니다.

## 목표 (Goal)

본격적인 구현에 앞서 핵심 산출물 3종(`spec.md`, `plan.md`, `tasks.md`) 간의 불일치, 중복, 모호함 및 불충분하게 명세된 항목들을 조기 식별합니다. 이 명령어는 **반드시** `/speckit.tasks`를 통해 완료된 `tasks.md`가 정상적으로 생성된 이후에만 실행되어야 합니다.

## 운영 제약 사항 (Operating Constraints)

**엄격한 읽기 전용 (STRICTLY READ-ONLY)**: 어떠한 문서 파일도 직접 수정하지 **마십시오**. 구조화된 분석 보고서를 결과로 출력합니다. (수동 후속 조치를 위해 사용자의 명시적인 사전 승인이 있을 경우에 한정하여) 선택적인 교정(remediation) 계획안을 제공할 수 있습니다.

**헌법의 권위 (Constitution Authority)**: 분석 범위 내에서 프로젝트 헌법(`.specify/memory/constitution.md`)의 내용은 **타협 불가** 합니다. 헌법과 충돌하는 내용은 자동으로 '치명적(CRITICAL)' 위반으로 간주되며 spec, plan, 혹은 tasks의 수정을 요구합니다—결코 헌법 원칙 자체를 축소 해석하거나 조용히 묵인해서는 안 됩니다. 만약 헌법의 원칙 자체가 변경되어야 한다고 판단되면, 이는 `/speckit.analyze` 외부에서 별도의 명시적인 헌법 업데이트 절차를 통해 이루어져야 합니다.

## 실행 단계 (Execution Steps)

### 1. 분석 컨텍스트 초기화

저장소 루트에서 `.specify/scripts/powershell/check-prerequisites.ps1 -Json -RequireTasks -IncludeTasks` 명령어를 한 번 실행하고 JSON 출력 결과를 파싱하여 FEATURE_DIR 경로와 AVAILABLE_DOCS 뷰를 확보합니다. 이를 기반으로 절대 경로를 도출하세요:

- SPEC = FEATURE_DIR/spec.md
- PLAN = FEATURE_DIR/plan.md
- TASKS = FEATURE_DIR/tasks.md

만약 필요한 파일 중 하나라도 누락된 경우, 사용자에게 어떤 선행 명령어를 실행해야 하는지 안내하는 에러 메시지와 함께 분석을 즉시 중단하십시오.
"I'm Groot"와 같이 작은따옴표가 포함된 문자열을 매개변수로 넘길 땐 이스케이프 구문(e.g., 'I'\''m Groot') 또는 가능하면 큰따옴표(e.g., "I'm Groot")를 사용하세요.

### 2. 산출물 로드 (점진적 정보 공개)

각 산출물에서 분석에 필수적인 최소한의 컨텍스트만 로드하십시오:

**spec.md 에서:**

- 개요/컨텍스트 (Overview/Context)
- 기능적 요구 사항 (Functional Requirements)
- 성공 기준 (측정 가능한 성과 — 예: 성능, 보안, 가용성, 사용자 작업 성공률, 비즈니스 지표 등)
- 사용자 스토리 (User Stories)
- 엣지 케이스 (존재하는 경우)

**plan.md 에서:**

- 아키텍처/기술 스택 선택 사항
- 데이터 모델 참조
- 개발 단계 (Phases)
- 기술적 제약 사항

**tasks.md 에서:**

- Task ID
- 각 태스크 설명
- 단계별(Phase) 그룹화
- 병렬 마커 [P]
- 참조된 파일 경로

**헌법(constitution) 에서:**

- 원칙 위반 여부 검증을 위해 `.specify/memory/constitution.md` 로드

### 3. 의미론적 모델 구축 (Build Semantic Models)

분석을 위한 내부 정보 모델을 구성하십시오 (출력 결과에는 raw 텍스트 파일 내용을 그대로 포함하지 마세요):

- **요구 사항 인벤토리**: 각 기능적 요구사항(FR-###)과 성공 기준(SC-###)에 대해 고유 키(key)를 기록합니다. 명시적인 FR-/SC- 식별자가 존재하는 경우 이를 기본 키로 사용하며, 추가적으로 가독성을 높이기 위해 짧은 요약 문구 슬러그(slug) 패턴을 생성할 수 있습니다(예: "User can upload file" → `user-can-upload-file`). 성능/보안 인프라 구축 등 실제 코드로 만들어 내야 하는(buildable) 성공 기준만 포함시키고, 출시 후 결과물 지표나 비즈니스 KPI(예: "고객 지원 티켓 접수율 50% 감소")는 제외합니다.
- **사용자 스토리/액션 인벤토리**: 인수 조건(acceptance criteria)이 포함된 개별 사용자 액션.
- **태스크 커버리지 맵핑**: 각 태스크를 하나 또는 그 이상의 요구사항/스토리에 맵핑합니다 (ID 같은 명시적 참조 패턴이나 키워드 문맥을 통한 추론 사용).
- **헌법 규칙 세트**: 헌법에서 원칙의 이름과 반드시(MUST)/권장(SHOULD)하는 규범적 문장을 추출합니다.

### 4. 분석 감지 패스 (토큰 효율적인 분석)

높은 신호(High-signal)를 가진 결과물에 집중하십시오. 분석 결과는 최대 50개 항목으로 제한하고, 초과분은 요약본에 합산합니다.

#### A. 중복 감지 (Duplication Detection)

- 거의 일치하거나 중복되는 요구 사항 식별
- 통합 병합을 위해 상대적으로 품질이 낮은 표현식 표기

#### B. 모호성 감지 (Ambiguity Detection)

- 측정 가능한 기준이 안 보이는 모호한 형용사(빠른, 확장 가능한, 안전한, 직관적인, 견고한 등) 지적
- 미해결 상태의 플레이스홀더 (TODO, TKTK, ???, `<placeholder>` 등) 지적

#### C. 불충분 명세 (Underspecification)

- 동사는 존재하지만 대상 목적어나 측정 가능한 결과가 누락된 요구 사항
- 인수 조건(acceptance criteria) 정렬이 누락된 사용자 스토리
- spec이나 plan에 정의되지 않은 파일이나 컴포넌트를 참조하고 있는 태스크

#### D. 헌법 준수 (Constitution Alignment)

- 헌법의 MUST 원칙을 위반하는 모든 요구 사항 또는 구현 계획(plan) 요소
- 헌법에서 요구하는 필수 입력 섹션이나 필수 품질 검증 게이트가 누락된 경우

#### E. 커버리지 공백 (Coverage Gaps)

- 연관된 태스크가 단 한 개도 없는 요구 사항
- 어떠한 요구 사항/스토리에도 맵핑되지 않은 태스크
- 코드로 작성되어야 할 성공 기준(성능, 보안, 가용성 보장 등) 중 tasks.md 에 전혀 반영되지 않은 항목

#### F. 불일치 (Inconsistency)

- 용어 표류(Terminology drift): 파일들 사이에서 동일한 개념이 다른 단어로 명명된 경우
- 데이터 엔티티 불일치: plan에는 나오지만 spec 에는 없는 경우 (혹은 그 반대)
- 태스크 순서 모순: 기반 환경 설정 태스크가 완료되기도 전에 의존성 관계 안내 없이 통합/연계 태스크가 앞으로 온 경우
- 상충되는 요구 사항 (예: 하나는 Next.js를 요구하는데 다른 요구사항은 Vue를 쓰는 것으로 명시된 경우)

### 5. 심각도 지정 (Severity Assignment)

다음 휴리스틱에 따라 각 발견 항목의 심각도를 지정하십시오:

- **원칙 위반 (CRITICAL)**: 기본 동작을 가로막는 요소이거나 주요 spec 아티팩트의 누락, 헌법 MUST 원칙 위반, 커버리지 맵핑 0건 요구사항 존재
- **위험 (HIGH)**: 중복 또는 상충되는 요구 사항, 측정이 불가한 애매한 성능/보안 속성명시, 테스트를 수행할 수 없는 인수 조건
- **경고 (MEDIUM)**: 용어 표류 현상, 비기능적 요구 태스크(Non-functional) 항목 누락, 덜 구체적인 엣지 케이스 명세
- **단순 권고 (LOW)**: 말투나 어조 개선, 작업 수행 순서에 영향을 미치지 않는 단순 중복

## 운영 기본 원칙 (Operating Principles)

### 컨텍스트 검토 시 효율성 주의

- **핵심 정보만 포함하는 출력물**: 불필요하게 문서를 구구절절 나열하지 말고 실질적으로 실행 가능한 결과 요소에 초점을 맞출 것.
- **점진적 공개 원칙**: 분석을 시작하자마자 파일의 처음부터 끝 텍스트 덤프를 올리지 말고 문서의 일부를 점진적으로 불러들일 것.
- **토큰 최적화**: 분석 테이블 결괏값 목록은 50행으로 제한하고 이를 초과하는 부분은 넘겨 버리고 요약 문구로만 압축할 것.
- **결정론적 일관성 보장**: 코드 변경이 없는 상태로 리포트를 재실행했을 때 항상 동일한 결과 ID, 통계 수치가 산출되어야 합니다.

### 분석 가이드라인

- **절대로 파일을 직접 수정하지 마십시오** (이 명령어는 오로지 읽기 전용 분석기입니다).
- **존재하지 않는 섹션을 억지로 짜내 상상하지 마십시오** (섹션이 비었다면 누락된 사실 그 자체를 있는 그대로 보고하세요).
- **헌법 위반 여부를 무엇보다 가장 최우선으로 검토하십시오** (이 유형은 무조건 CRITICAL 로 봅니다).
- **포괄적 설명 위주의 규정 언급보다는 실제 사례 위주로 지적하십시오** (일반화 시키지 말고 구체적인 인스턴스를 인용하십시오).
- **이슈가 0개 관측될 때도 당당하게 대응하십시오** (커버리지 통계와 함께 문제없음 상태를 밝은 어조로 출력하세요).
