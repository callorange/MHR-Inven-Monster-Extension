## 사용자 입력 (User Input)

```text
$ARGUMENTS
```

진행하기 전에 사용자 입력값이 있다면 **반드시** 이를 고려해야 합니다.

## 사전 실행 확인 (Pre-Execution Checks)

**확장 훅(Extension Hooks) 확인 (명세화 단계 이전)**:
- 프로젝트 루트에 `.specify/extensions.yml` 파일이 존재하는지 확인합니다.
- 존재한다면 파일을 읽어 `hooks.before_specify` 키 아래의 항목들을 찾습니다.
- YAML 파싱이 실패하거나 유효하지 않은 경우, 훅 확인 과정을 조용히 건너뛰고 정상적으로 계속 진행합니다.
- `enabled`가 명시적으로 `false`인 훅은 제외합니다. `enabled` 필드가 없는 훅은 기본적으로 활성화된 것으로 간주합니다.
- 남은 훅들에 대해 `condition` 표현식을 직접 해석하거나 평가하려고 시도하지 **마십시오**:
  - 훅에 `condition` 필드가 없거나 비어 있는 경우, 해당 훅을 실행 가능한 것으로 간주합니다.
  - 훅에 내용이 있는 `condition`이 정의된 경우, 해당 훅을 건너뛰고 조건 평가는 HookExecutor 구현체에 맡깁니다.
- 실행 가능한 각 훅에 대해 `optional` 플래그에 따라 다음 중 하나를 출력합니다:
  - **선택적 훅** (`optional: true`):
    ```
    ## 확장 훅 (Extension Hooks)

    **선택적 사전 훅(Optional Pre-Hook)**: {extension}
    명령어: `/{command}`
    설명: {description}

    프롬프트: {prompt}
    실행하려면: `/{command}`
    ```
  - **필수 훅** (`optional: false`):
    ```
    ## 확장 훅 (Extension Hooks)

    **자동 사전 훅(Automatic Pre-Hook)**: {extension}
    실행 중: `/{command}`
    EXECUTE_COMMAND: {command}

    Outline 단계로 넘어가기 전에 이 훅 명령어의 수행 결과를 기다리십시오.
    ```
- 등록된 훅이 없거나 `.specify/extensions.yml`이 존재하지 않으면 조용히 건너뜁니다.

## 개요 (Outline)

사용자가 `/speckit.specify` 명령어 뒤에 입력한 텍스트가 **곧 기능 설명(feature description)** 입니다. 만약 시스템 변수 `$ARGUMENTS` 가 아래쪽에 곧이곧대로 보이더라도, 이미 대화상에서 해당 텍스트를 제공받았다고 간주하십시오. 사용자가 완전히 빈 명령어를 제공한 것이 아니라면, 다시 기능을 설명해 달라고 묻지 마십시오.

해당 기능 설명을 바탕으로 다음을 수행하십시오:

1. **간결한 짧은 브랜치 이름 생성** (2~4 단어 길이):
   - 기능 설명을 분석하여 가장 의미 있는 키워드를 추출합니다.
   - 기능의 본질을 잘 포착하는 2~4 단어 길이의 약식 이름(short name)을 만듭니다.
   - 가급적 작동-명사(action-noun) 포맷을 사용하십시오 (예: "add-user-auth", "fix-payment-bug").
   - 기술 전문 용어나 약어(OAuth2, API, JWT 등)는 그대로 보존하십시오.
   - 간결하면서도 한눈에 이 기능이 무엇인지 파악할 수 있도록 서술성을 띄게 지으십시오.
   - 예시:
     - "사용자 인증 기능을 추가하고 싶습니다" → "user-auth"
     - "API용 OAuth2 통합 연동 구현" → "oauth2-api-integration"
     - "분석용 대시보드 생성" → "analytics-dashboard"
     - "결제 대기 시간 초과 버그 수정" → "fix-payment-timeout"

2. **기능 브랜치(feature branch) 생성**: `.specify/scripts/powershell/create-new-feature.ps1` 스크립트를 `--short-name` 매개변수(그리고 `--json`)와 함께 실행하십시오. (순차 번호 지정 모드인 sequential 모드의 경우, `--number` 옵션을 수동으로 넘기지 마십시오 — 스크립트가 알아서 다음 사용 가능한 번호를 찾아 할당합니다. 타임스탬프 모드일 경우 `YYYYMMDD-HHMMSS` 접두어를 알아서 만듭니다.)

   **브랜치 번호 지정 모드 (Branch numbering mode)**: 스크립트를 실행하기 전, `.specify/init-options.json` 파일이 존재하는지 확인하고 내부의 `branch_numbering` 값을 읽으십시오.
   - 값이 `"timestamp"` 일 경우, 스크립트 실행 시 `--timestamp` 옵션을 추가하십시오 (-Timestamp 또는 --timestamp).
   - 값이 `"sequential"` 이거나 아예 존재하지 않는다면 기본 동작으로 간주해 추가 옵션을 넣지 마십시오.

   - 예시 (명령어 구문 구조 - Bash & PowerShell 대응):
     - Bash: `.specify/scripts/powershell/create-new-feature.ps1 "$ARGUMENTS" --json --short-name "user-auth" "Add user authentication"`
     - Bash (타임스탬프): `.specify/scripts/powershell/create-new-feature.ps1 "$ARGUMENTS" --json --timestamp --short-name "user-auth" "Add user authentication"`
     - PowerShell: `.specify/scripts/powershell/create-new-feature.ps1 "$ARGUMENTS" -Json -ShortName "user-auth" "Add user authentication"`
     - PowerShell (타임스탬프): `.specify/scripts/powershell/create-new-feature.ps1 "$ARGUMENTS" -Json -Timestamp -ShortName "user-auth" "Add user authentication"`

   **중요 (IMPORTANT)**:
   - `--number` 옵션을 넘기지 마십시오 — 이 스크립트는 다음 올바른 숫자를 자동 계산합니다.
   - 반드시 JSON 플래그(`--json` 또는 `-Json`)를 포함하여, 결괏값을 신뢰할 수 있는 형태로 파싱할 수 있게 만듭니다.
   - 이 스크립트는 단일 기능 생성 시 딱 한 번만 실행해야 합니다.
   - 스크립트 실행 결과로 생성된 JSON 포맷 텍스트가 터미널에 출력됩니다. 이를 직접 읽고 찾고자 했던 실젯값(생성된 디렉토리 등)을 확인하십시오.
   - 해당 JSON 출력물에는 `BRANCH_NAME` 과 `SPEC_FILE` 의 확정된 경로 정보가 담겨 있습니다.
   - "I'm Groot"처럼 단일 따옴표가 파라미터 내부에 섞여있다면 이스케이프 구문 ('I'\''m Groot') 이나 쌍따옴표 구문("I'm Groot") 구조를 활용하여 파라미터 파괴 현상을 회피하십시오.

3. `.specify/templates/spec-template.md` 템플릿을 읽어 필수 작성 섹션들을 파악합니다.

4. 이하의 실행 절차를 따릅니다:

    1. 입력(Input)에서 사용자의 기능 설명 텍스트를 파싱합니다.
       설명이 비었다면: 에러 방출 -> ERROR "기능에 대한 설명이 제공되지 않았습니다 (No feature description provided)"
    2. 설명에서 핵심 컨셉(개념)을 추출해 냅니다.
       식별 타겟: 액터(배우/사용자), 액션(동작), 처리될 데이터, 한계/제약조건.
    3. 명확하지 않고 붕 뜨는 대목들이 있다면:
       - 문맥과 업계 표준 원칙에 입각해 지능적인 추측(informed guesses)을 시도합니다.
       - 정말 어쩔 수 없을 때에만 `[NEEDS CLARIFICATION: 구체적인 질문]` 마커를 표시하되 다음의 경우에 한정하십시오:
         - 이 대안 선택이 향후 진행될 기능 스코프 범위나 UX에 막대한 지장을 초래할 때
         - 여러 갈래로 해석될 수 있으면서 각 방향마다 파급 효과가 완전히 다를 때
         - 채택할 만한 상식적인 기본값(default) 설정이 불가능할 때
       - **제한 설정: [NEEDS CLARIFICATION] 마커는 전체 문서를 통틀어 최대 3개까지만 달 수 있습니다.**
       - 우선순위 기준: 스코프의 크기 결정 > 보안/프라이버시 > 사용자 경험(UX) > (가장 낮은 순위) 단순 기술 디테일 사항.
    4. 사용자 시나리오 및 테스팅(User Scenarios & Testing) 섹션을 채웁니다.
       어떠한 명확한 유저 플로우도 그려지지 않는다면: 에러 방출 -> ERROR "사용자 시나리오를 특정할 수 없습니다 (Cannot determine user scenarios)"
    5. 기능 요구 사항(Functional Requirements)을 생성합니다.
       각 요구사항은 어떻게든 테스트 가능(testable)해야 합니다.
       세부 사항이 명시되지 않았다면 합리적 기본값을 상정하여 채우십시오 (그리고 이걸 전제조건 영역에 기재하십시오).
    6. 성공 평가 기준(Success Criteria)을 정의합니다.
       측정 가능하고, 특정 언어나 기술 등에 얽매이지 않는 결과 지향적 지표를 창조하십시오.
       정량적 지표(소요 시간, 퍼포먼스 수치, 볼륨) 및 정성적 지표(유저 만족도, 태스크 완수율)를 모두 취합하세요.
       각 기준점은 나중에 "어떻게 구현했건 간에" 이를 확인할 수 있어야 합니다.
    7. 다뤄야 하는 데이터가 있다면 핵심 엔티티 구조를 파악합니다.
       완료 시명세서가 이제 플래닝(plan) 단계로 넘어갈 준비가 되었음을 선언합니다: SUCCESS (spec ready for planning)

5. 앞서 파싱해둔 `SPEC_FILE` 경로에 완성된 사양 명세서를 파일로 기록하여 씁니다. 이때 템플릿 구조를 엄격히 지켜 모든 섹션 정렬과 헤딩 포맷을 보존하고, 빈 플레이스홀더를 사용자가 요구한 실질 내용으로 알차게 포장 교체해야 합니다.

6. **명세서 품질 검증 (Specification Quality Validation)**: 초기 명세서를 작성하고 난 뒤, 아래의 품질 기준에 부합하는지 즉시 검증 패스를 진행하십시오:

   a. **명세서 품질 자체 체크리스트 생성**: 다음과 같은 항목이 담긴 체크리스트 평가표를 `FEATURE_DIR/checklists/requirements.md` 경로에 생성하십시오:

      ```markdown
      # 기능 명세서 품질 체크리스트: [기능 이름 FEATURE NAME]
      
      **목적(Purpose)**: 설계 단계(plan)에 들어가기 전 명세서의 내용 완성도와 품질 규격을 검증하기 위함입니다.
      **작성일(Created)**: [DATE]
      **관련 기능 명세(Feature)**: [생성된 spec.md 링크]
      
      ## 콘텐츠 품질 (Content Quality)
      
      - [ ] 구체적 구현 코드 디테일(언어명, 프레임워크명, 특정 API 이름)이 포함되지 않았는가?
      - [ ] 기술 구현보다는 사용자가 얻을 가치와 비즈니스 요구에 초점을 두었는가?
      - [ ] 비-기술자(non-technical) 이해관계자들도 읽고 이해할 수 있는 언어로 쓰였는가?
      - [ ] 템플릿 내의 모든 필수 작성 섹션들이 채워졌는가?
      
      ## 요구조건의 완결성 (Requirement Completeness)
      
      - [ ] 더 이상 명확화를 요구하는 [NEEDS CLARIFICATION] 임시 마커가 남아있지 않은가?
      - [ ] 요구된 조건들이 테스트/계측 가능하며, 모호한 표현 없이 명료하게 기재됐는가?
      - [ ] 성공 기준(Success criteria)이 측정 가능한 수치나 명쾌한 달성 조건으로 표기됐는가?
      - [ ] 성공 기준에 특정 스택이나 기술 종속적인 단어가 배제되었는가? (technology-agnostic)
      - [ ] 모든 인수를 위한 시나리오 플로우가 정의되었는가?
      - [ ] 예외 처리 및 엣지 케이스 시나리오가 식별되었는가?
      - [ ] 구현 스코프의 바운더리(한계선)가 분명히 쳐져서 명시되었는가?
      - [ ] 종속성(Dependencies) 모델과 가정(assumptions)들이 누락 없이 기록되었는가?
      
      ## 기능 배포 준비도 (Feature Readiness)
      
      - [ ] 모든 기능 작동 요건들이 그에 상응하는 명확한 인수 조건(Acceptance criteria)을 가졌는가?
      - [ ] 제시된 유저 시나리오가 주요 경로(primary flows) 로직을 전부 담고 있는가?
      - [ ] 성공 기준에 작성된 측정 타겟을 이 기능이 완벽히 만족시킬 수 있게 쓰였는가?
      - [ ] 어떠한 구현/소프트웨어 공학적 상세 내역도 명세서 내용에 흘러나오지(leak) 않았는가?
      
      ## 노트 (Notes)
      
      - 아직 미완료 처리된 위 문항들이 해소되기 전까진 `/speckit.clarify`나 `/speckit.plan`으로 넘어가선 안 됩니다.
      ```

   b. **유효성 검증(Validation) 수행**: 위 체크리스트 문항에 비추어본 당신이 작성한 최신 spec 파일을 자체 대조 점검하십시오:
      - 각 문항이 통과(Pass)할 수 있는지 혹은 실패(Fail)인지 냉정하게 판단하십시오.
      - 발견된 특정 결함들을 짚어내 내역에 문서화하십시오 (명세서 항목 중 어떤 대목이 문제가 되는지 인용 첨부).

   c. **유효성 검증 결과물 핸들링 (Handle Validation Results)**:

      - **모든 항목 통과 시 (If all items pass)**: 체크리스트를 완료(체크) 표시 한 다음 단계 7로 나아가십시오.

      - **안타깝게도 실패 항목 위반 시 ([NEEDS CLARIFICATION] 마커를 제외하고 구멍이 난 경우)**:
        1. 깨진 항목들과 해당 요소들을 열거하십시오.
        2. 위 이슈들을 수렴해 spec 파일 본문을 직접 교정 업데이트 하십시오.
        3. 모두 통과할 때까지 재점검 루프를 돕니다 (단, 무한 루프 방지를 위해 이 짓은 최대 3회로 자동 강제 제한합니다).
        4. 3번이나 스펙을 다시 고쳐썼는데도 여전히 기준점 미달이라면 남은 구멍 난 사항을 checklist notes 파트에 경고문으로 적어놓고 사용자에게 이를 알립니다.

      - **만약 [NEEDS CLARIFICATION] 질문 마커가 문서 본문에 방치된 채 남아있을 경우**:
        1. spec 파일 내에 포진한 `[NEEDS CLARIFICATION: ...]` 구문들을 모조리 수거하십시오.
        2. **제한 초과 감시**: 만일 마커가 3개를 돌파하여 왕창 생겼다면, 가장 치명적인 영향도를 가진(스코프/보안/UX) 상위 3개 질문 마커만 남기고 나머지 마커 파트들은 당신이 알아서 지능적인 상상력을 발휘해 추론하여 임의로 문장을 채워 덮어버리십시오.
        3. 살아남은 (최대 3개의) 필요 질문 사항들에 대하여, 사용자에게 답을 얻어내기 위해 다음과 같은 양식으로 화면에 선택지를 띄우십시오:

           ```markdown
           ## 추가 확인 질문 [N]: [해당 토픽명]
           
           **배경 맥락(Context)**: [명세서 내부 질문 유발 상황 인용구 텍스트]
           
           **도움이 필요한 포인트**: [실제 작성했던 NEEDS CLARIFICATION 마커 내용]
           
           **추천 답변 방향 구상**:
           
           | 선택옵션 | 적용할 내용 | 해당 옵션을 골랐을 때 미치는 파급효과 |
           |--------|--------|--------------|
           | A      | [첫 번째 제안 시나리오] | [분기 A로 갔을 시 그려질 이 기능의 운명] |
           | B      | [두 번째 제안 시나리오] | [분기 B로 갔을 시 그려질 이 기능의 운명] |
           | C      | [세 번째 제안 시나리오] | [분기 C로 갔을 시 그려질 이 기능의 운명] |
           | 직접작성 | 사용자님의 독자적인 생각 기입 | [이 기능이 구현될 수 있는 형태라면 무엇이든] |
           
           **사용자님의 선택**: _[질문 멈추고 사용자 응답 대기]_
           ```

        4. **치명적 주의 - 표 양식 관련 (CRITICAL - Table Formatting)**: 이쁘고 정상적인 마크다운 구문을 지킬 것:
           - 열 나누는 파이프(`|`) 기호가 일정하고 정돈된 간격을 지킬 것.
           - 각 셀 내용물 좌우로 공백이 한 칸씩 있을 것: `|Content|` 가 아니라 `| Content |`
           - 헤더 라인과 콘텐츠 라인을 찢는 분리 줄에 반드시 대시(`-`)를 3개 이상 투자할 것: `|---|` 혹은 `|--------|`
           - 마크다운 렌더러가 박살 나지 않을지 다시 점검해 볼 것.
        5. 여러 질문이 나왔다면 질문 넘버링을 붙이십시오 (Q1, Q2, Q3 등 - 최대 3개).
        6. 입만 다물고 기다리지 말고 일단 모아둔 질문들을 사용자 뷰에 다 토해 내어 보여준 다음 대기 상태에 진입하십시오.
        7. 사용자가 입력할 답변(예: "Q1: A로 가자, Q2: 내가 적어본다 - [세부 내용], Q3: B")을 기다리십시오.
        8. 받은 답변 내용으로 spec 내 남아있던 흉측한 `[NEEDS CLARIFICATION]` 마커 부위를 뜯어내고 말끔한 내용으로 치환 교체 업데이트 하십시오.
        9. 이제 명확하지 않은 부분이 다 풀렸으므로, 전체 유효성 재검증 패스(Validation pass)를 다시 한 번 더 태우십시오.

   d. **구조 점검표 갱신 (Update Checklist)**: 검증 이터레이션이 1회 종료될 때마다 체크리스트 파일 안의 통과/실패(pass/fail) 상태값을 최신화해서 맞추십시오.

7. 처리 완료 상황을 당신의 보고서로 마감합니다: 사용한 브랜치 이름 표시, spec 절대 경로 위치 표시, 검증 체크리스트 결과 요약본 전송, 그리고 다음 단계를 위해 (`/speckit.clarify`로 갈 건지 `/speckit.plan`으로 갈 건지) 준비가 되었음을 선언합니다.

8. **확장 훅(Extension Hooks) 확인 (완료 이후)**: 보고서 출력 후, सिस्टम 루트에 `.specify/extensions.yml` 파일이 있는지 확인합니다.
   - 존재한다면 파일을 읽어 `hooks.after_specify` 키 아래의 항목들을 찾습니다.
   - YAML 파싱이 실패하거나 유효하지 않은 경우, 훅 확인 과정을 조용히 건너뛰고 정상 진행합니다.
   - `enabled`가 명시적으로 `false`인 훅은 제외합니다. 명시되지 않았다면 활성화된 것으로 간주합니다.
   - 남은 훅들에 대해 `condition` 표현식의 자체 직접 평가를 피하십시오. 그건 HookExecutor가 다루게 놔두고 즉시 실행 가능 상태로 패스시킵니다.
   - 실행 가능한 각 훅에 대해 `optional` 에 따라 다음을 출력합니다:
     - **선택적 훅** (`optional: true`):
       ```
       ## 확장 훅 (Extension Hooks)

       **선택적 훅(Optional Hook)**: {extension}
       명령어: `/{command}`
       설명: {description}

       프롬프트: {prompt}
       실행하려면: `/{command}`
       ```
     - **필수 훅** (`optional: false`):
       ```
       ## 확장 훅 (Extension Hooks)

       **자동 훅(Automatic Hook)**: {extension}
       실행 중: `/{command}`
       EXECUTE_COMMAND: {command}
       ```
   - 등록된 훅이 없거나 `.specify/extensions.yml` 파일 자체가 없다면 조용히 무시하고 마무리 짓습니다.

**참고 (NOTE):** powershell 기반 파생 스크립트는 새로운 브랜치를 생성하고 그곳으로 자동으로 체크아웃 시켜주며 쓰기 작업 직전에 기능 명세서 파일 뼈대를 초기화해 놓는 역할까지 수행합니다.

## 빠른 가이드라인 (Quick Guidelines)

- 사용자가 **무엇(WHAT)**을 바라는지, 도대체 **왜(WHY)** 이걸 만들고 싶어 하는지에 강박적으로 매달리십시오.
- 어떻게 이 기능을 프로그래밍 짜서 구현할 것인지(HOW)에 대해선 생각하지 마십시오 (기술 스택, API, 코드 구조 등 기재 불가).
- 철저히 엔지니어가 아닌 현업 담당자(Business stakeholders)가 술술 읽기 쉬운 언어로 써 내려가십시오.
- spec 명세서 안에다가 덤으로 온갖 체크리스트를 같이 욱여넣는 만행을 피하십시오. 체크리스트는 별도의 파일 관리 커맨드로 존재합니다.

### 템플릿 섹션 기입 룰 (Section Requirements)

- **필수 대단원 (Mandatory sections)**: 피칠갑을 해서라도 모든 기능 파트에 기입을 강제해야 하는 핵심 기둥들.
- **옵션 단원 (Optional sections)**: 이 특정 기능과 결이 맞는 구석이 있을 때에만 선택해서 넣는 파트들.
- 해당 단원이 아예 쓸모가 없다 판단될 땐, 대놓고 섹션을 도려내십시오 (어설프게 "해당 없음(N/A)" 이라고 텍스트 박아놓고 남겨두지 마세요).

### AI 문장 생성 에이전트 행동 지침 (For AI Generation)

사용자의 한 줄 프롬프트를 웅장한 spec 으로 펑튀기 창조해 낼 때:

1. **똑똑한 짐작하기 (Make informed guesses)**: 주워진 문서 컨텍스트 동향, 코딩 업계 표준 관례, 보편적으로 취하는 디자인 패턴 방식을 참고해서 비어있는 여백의 정보 공백을 꽉곽 채우십시오.
2. **도입 가정 명시하기 (Document assumptions)**: 암묵적으로 적용한 상식적 기본값들이 있다면 가설/가정(Assumptions) 섹션 단원에 "이러이러하게 작동한다고 가정함" 이라고 못 박아 기록하십시오.
3. **질문 억제 (Limit clarifications)**: 확인차 묻기 마커(`[NEEDS CLARIFICATION]`)는 3개까지만 달 수 있습니다. 그것도 다음과 같은 상황에서만 쓰십시오:
   - 이걸 어느 쪽으로 선택하냐에 따라 스코프 범위 배수진이 쳐지거나 UX가 하늘과 땅 차이로 벌어질 때.
   - 다양한 해석의 여지가 다분하여 어느 방향이건 설득력이 높은 양립성이 존재할 때.
   - 보편 타당한 '기본값' 따위로 퉁 치고 넘길 수 없는 문제일 때.
4. **질문 우선순위 배정 (Prioritize clarifications)**: (가장 중요) 구현 범위 확정(scope) > 보안 및 보호 수단 > 사용자 체감 디자인(UX) > 기능 동작 디테일 구현.
5. **테스터처럼 자아 의탁하기 (Think like a tester)**: 문서에 적은 문구가 자기가 봐도 뜬구름 잡는 소리 같다면, '테스트할 수 있으며 더 이상 모호함 없는' 품질 기준 체크리스트 테스트에 불합격표(`FAIL`)를 던지십시오.
6. **마땅한 정답이 없어 가장 질문이 많이 터져 나오는 대표 영역 패턴들 (Common areas needing clarification)** (오직 믿고 박을 상식적 기본값이 전무할 때만):
   - 기능 도입 경계 및 한계 차단벽 (어떤 유형의 고객 이용 케이스를 허용하고/차단할 것인가)
   - 권한 분리와 유저 계층 등급 (해석에 따라 열람 권한이 완전히 뒤바뀔 소지가 농후할 때)
   - 보안 / 법령 준수 준법성 요건 (금전적 피해나 구속 등 법적 논란 소지가 다분한 섹터일 경우)

**이런 걸 굳이 사용자에게 여쭤보지 마라 (Examples of reasonable defaults - don't ask about these)**:
(다 알아서 상식적으로 통용되는 기본값들)

- 데이터 보존 기한 설정: 그냥 해당 상업 도메인의 일반적인 표준 관례 데이터 보존 기간을 따른다고 명시할 것.
- 성능 처리 기준 속도: 느려터진다고 특별히 명시된 사항이 없는 한 일반적인 웹/앱 기준 레이턴시와 로딩 기대치를 기준으로 작성할 것.
- 에러 상황 표시: 사용자 친화적인 안내 메시지를 뿜고 대체 가능한 fallback 경로로 이어진다는 수준으로 적을 것.
- 로그인 수단: 일반적인 웹 서비스 수준의 세션 기반 로그인 또는 OAuth2 토큰을 적용.
- 외부 의존성 통합 아키텍처 패턴: 그 서비스에 맞는 패턴(웹은 REST/GraphQL, 로컬 개발 툴은 CLI 인자 구조나 함수 바이너리 콜 등)을 자연스레 적용.

### 성공 지표 설정 세부 기준 규칙 (Success Criteria Guidelines)

단원명이 Success criteria라면 반드시 다음 특성을 내포해야 합니다:

1. **측정 가능성(Measurable)**: 지표 메트릭 단위 포함 필수 (기울인 시간, % 퍼센트, 처리 카운트 양, 변동 비율 등).
2. **기술 종속성 회피(Technology-agnostic)**: 어떠한 경우에도 프레임워크나 언어명, 시스템 데이터베이스 이름이나 특수한 툴 패키지 등이 등판해선 안 됨.
3. **이용자 포커싱(User-focused)**: 시스템 내부 사정이 어떻건 간에 사용자와 비즈니스 모델 입장에서 체감되는 결과물(outcomes)만을 적어 놓을 것.
4. **증명 가능성(Verifiable)**: 개발자가 내부 코드를 어떻게 더럽게 짜놨건, 테스트 팀은 외부에서도 구현 디테일 없이 그 지표의 거짓 유무를 증명/검증할 수 있어야 함.

**아주 훌륭한 예시 모음 (Good examples)**:

- "일반 사용자는 결제 절차 폼을 3분 이내에 모두 통과하여 완료할 수 있다."
- "이 시스템은 동시에 만 명의 사용자가 접속하여 조작해도 뻗지 않는다."
- "유저가 검색을 때리면 95% 확률로 1초 언더의 쾌적한 속도로 결과 페이지가 로딩된다."
- "특정 태스크 도달/성공 확률(완수율)이 예전보다 40% 이상 수직 상승한다."

**형편없는 예시 모음 - 기술 구현 디테일에 쩔여진 상태 (Bad examples)**:

- "API 리스폰스 반환 타임 속도가 200ms를 초과하지 않는다." (너무 공학적입니다. -> "사용자가 버튼 누르면 체감상 즉시 화면이 전환된다" 로 수정 요망)
- "데이터베이스가 TPS 1000 쿼리 처리를 감당할 수 있다." (내부 시스템 사정입니다. -> 화면을 보는 사용자 관점 메트릭으로 돌려쓸 것)
- "React 컴포넌트가 최신 VDOM에 의해 극도로 효율적으로 리렌더링 된다." (어떤 프레임워크인지 언급해 버렸습니다.)
- "Redis 메모리 캐시 서버 Hit 적중률(캐시히트)이 항상 80% 상한선을 유지한다." (어떤 기술 스택을 사용하는지 대놓고 발언해버렸습니다.)
