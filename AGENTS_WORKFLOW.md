# 🛠 Standard Operational Workflow (Modular v2.2)

모든 작업은 체계적인 3단계 프로세스를 **필수적으로 거쳐야 합니다.** 각 단계의 완료와 사용자의 승인을 확인한 후 다음 단계로 진행하십시오.

## Phase 1: Research (연구 및 분석)
- **심층 분석**: 관련 파일의 구조와 의존성을 분석하고 논리적 비약을 방지하기 위한 **단계별 계획(Step-by-step)**을 수립합니다.
- **파괴 가능성 검토**: 신규 기능 도입 시 기존 기능의 **파괴(Regression) 가능성을 최우선으로 검토**합니다.

## Phase 2: Strategy & Specification (전략 및 명세 확립)
- **최적안 도출**: 분석 결과를 바탕으로 기술적 타당성이 검증된 **가장 단순하고 명확한 방안을 도출**합니다.
- **검증 기준 사전 합의**: 코드를 작성하기 전, 요구사항을 충족하는지 확인할 수 있는 **명세(Spec)나 테스트 시나리오(Edge Case 포함)를 미리 도출**하여 사용자에게 제시합니다.
- **의사결정 승인**: 수립된 전략과 검증 기준을 설명하고 **최종 진행 승인을 획득**합니다.

## Phase 3: Execution & Sync (실행 및 철저한 검증)
- **정밀 구현**: 확립된 명세와 합의된 검증 기준을 통과할 수 있도록 코드를 **오차 없이 정확하게 구현**합니다. (방어적 코딩 준수)
- **안전한 에러 복구 (Error Recovery)**: 실행 단계에서 예기치 않은 오류가 발생했을 때, 임의로 코드를 롤백하거나 섣불리 수정하지 마십시오. 반드시 `1) 에러 로그 원문 분석 -> 2) 원인 가설 수립 -> 3) 해결 방안 제안`의 순서로 보고한 후 재승인을 얻어 수정합니다.
- **[💯 10점 만점 자체 평가 (Self-Correction)]**: 답변이나 코드를 최종 출력하기 전, 세계 최고 수준의 평가 기준을 스스로 설정하고 작성한 초안을 내부적으로 비판하십시오. *"이 답변에 논리적 비약이나 사실이 아닌 내용이 포함되어 있는가?"*를 엄격히 질문하고, 10점 만점을 받을 때까지 반복 개선한 후 출력해야 합니다.
- **완결성 테스트**: 사전에 합의한 검증 기준(Phase 2)의 충족 여부를 입증하며, 사용자에게 **최종 결과만 정제하여 보고**합니다.
- **[🔥 필수 엄수: 코드-산출물 100% 동기화]**: 실제 코딩을 진행하면서 아키텍처, DB 스키마, API 규약, 핵심 로직에 변경이 발생하면, **관련 산출물 문서를 반드시 찾아내어 코드와 완벽하게 동일한 최신 상태로 업데이트** 하십시오. 문서 업데이트가 누락된 작업은 완료된 것으로 인정하지 않습니다.

---
*정의된 절차의 준수가 결과물의 신뢰성을 보장합니다. 모든 단계를 충실히 이행하십시오.*
