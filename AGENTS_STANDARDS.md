# 📏 Coding & Commit Standards (v1.13)

## 1. 코딩 표준 (Coding Standards)
- 언어별 표준 스타일 가이드 및 프로젝트에 설정된 **Linter/Formatter 규칙을 엄격히 준수**합니다.
- 변수명 등 식별자는 영어를 사용하되, **모든 주석은 반드시 한국어를 사용**합니다.
- **[📐 구조적 문서화 (API Docs / Docstrings)]**: 패키지, 모듈, 클래스, 메서드, 함수 등 코드의 주요 구조적 단위 선언부에는 해당 언어 생태계의 공식 문서화 표준(예: Python Docstring, Javadoc, JSDoc 등)을 준수하여 역할, 파라미터, 반환값, 예외 사항을 명시하십시오.
- **[🔄 문맥 전환 주석 (Contextual Comments)]**: 함수나 메서드 내부에서 로직의 단계가 넘어가거나 복잡한 분기/계산이 시작되는 등 **'코드의 문맥이 전환되는 시점'**에는, 다음 블록의 목적을 안내하는 주석을 반드시 추가하여 가독성을 확보하십시오.
- **[🔥 Why 중심 주석 원칙]**: 위 모든 구조적 문서화와 문맥 주석을 작성할 때는 코드가 '무엇(What)'을 하는지 단순 번역하지 말고, **'왜(Why)' 이런 방식과 비즈니스 로직을 택했는지 배경 맥락과 의도를 서술**하십시오.
- **[🛡️ 관측 가능성 (Observability)]**: 에러나 중요 로직 분기점에서는 단순히 에러 메시지만 출력하지 마십시오. 운영 단계에서 원인을 즉시 추적할 수 있도록, **'어떤 상태(Context/Payload)'에서 실패했는지 로그에 반드시 포함**하십시오. (단, 민감 정보는 마스킹 처리)
- **[🛡️ 안전한 실패 (Fail-Fast & Graceful Degradation)]**: 유효하지 않은 상태나 잘못된 파라미터가 감지되면 조기에 명확한 예외를 던져 실행 흐름을 차단하십시오. 통제 불가능한 외부 연동 시에는 반드시 타임아웃과 예외 처리를 구현하여 시스템 전체 장애로 번지는 것을 막아야 합니다.
- **[🛡️ 보안 지향 설계 (Security by Design)]**: 외부로부터 들어오는 모든 입력값은 오염되었다고 가정하고 엄격히 검증(Validation)하며, 클라이언트에게 반환하는 API 에러 응답에는 내부 시스템 구조나 DB 쿼리문 등 민감한 정보가 절대 노출되지 않도록 마스킹하십시오.

## 2. 커밋 컨벤션 (Commit Convention)
- **Conventional Commits** 규칙을 엄격히 준수하여 의미 있는 히스토리를 관리합니다. (예: feat, fix, docs, style, refactor, test, chore)

## 3. 문서화 및 산출물 표준 (Documentation & Artifacts)
프로젝트의 안정적인 유지보수와 AI 에이전트의 완벽한 컨텍스트 파악을 위해, 핵심 산출물은 `docs/` 디렉토리 하위에 **주제별 모듈화 방식**으로 관리합니다.
모든 문서 작성 시 가독성 높은 마크다운을 사용하며, 단순 현상 나열이 아닌 **'기술적 의사결정의 근거(Why)'를 명확히 포함**해야 합니다. (작업 시 문서 동기화 의무는 `AGENTS_WORKFLOW.md`를 따릅니다.)

**[📂 표준 산출물 종류 및 정렬 규칙 (Numbering Prefix)]**
에이전트는 논리적인 읽기 흐름(Reading Flow)을 보장하기 위해, `docs/` 하위에 문서 생성 시 **반드시 두 자리 숫자 접두어**를 사용하여 파일명을 지정하고 아래의 목적/관리 지침을 준수합니다.

1. **`00_index.md` (진입점 및 목차)**
   - **목적:** 문서 전체의 구조를 조망하는 지도 역할이자, 에이전트의 컨텍스트 파악 시작점
2. **`01_architecture_design.md` (시스템 아키텍처)**
   - **목적:** 데이터 흐름, 인프라 구성, 모듈 간 연동 방식 정의
3. **`02_interface_contract.md` (인터페이스 및 도구 규약)**
   - **목적:** 프론트엔드 통신 API 또는 외부 연동(MCP Tools 등) 입출력 규약 명세
4. **`03_database_schema.md` (데이터베이스 스키마)**
   - **목적:** 테이블 구조, 인덱스 전략, 필드 타입 명세
5. **`04_[주제별]_strategy.md` (도메인 핵심 전략)**
   - **목적:** 프로젝트의 핵심이 되는 복잡한 비즈니스 알고리즘이나 전략 서술
6. **`99_tech_stack_rationale.md` (기술 스택 및 진화 이력)**
   - **목적:** 기술적 패러다임, 프레임워크 도입 및 변경 사유를 누적 기록 (Append-only)

**[🌱 문서 확장 전략 (Scaling Strategy)]**
- 단일 `docs/` 폴더 내의 마크다운 문서가 크게 증가하여 탐색이 어려워질 경우, 에이전트는 사용자에게 **카테고리별 하위 폴더 구조로의 전환을 제안**하고, 승인 시 문서를 체계적으로 마이그레이션해야 합니다.
- 추가적인 산출물이 필요하다고 판단될 경우, 에이전트는 위의 넘버링 규칙에 맞춰 스스로 적절한 마크다운 파일을 생성하여 문서를 확장하십시오.
