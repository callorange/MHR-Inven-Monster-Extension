# 태스크: [FEATURE NAME]

**입력값**: `/specs/[###-feature-name]/` 디렉토리의 설계 문서 리스트
**선행 조건 (Prerequisites)**: plan.md (필수), spec.md (사용자 스토리 식별에 필수), research.md, data-model.md, contracts/

**테스트**: 아래 예시에는 테스트 태스크가 포함되어 있습니다. 테스트는 선택 사항(OPTIONAL)이며, 기능 명세서에 명시적으로 요청된 경우에만 포함시키세요.

**구성**: 태스크는 각 스토리의 독립적인 구현 및 테스트를 위해 사용자 스토리 별로 그룹화되어 있습니다.

## 양식: `[ID] [P?] [Story] 설명`

- **[P]**: 병렬로 실행 가능한 항목 (파일 겹침 및 의존성 없음)
- **[Story]**: 이 태스크가 속한 사용자 스토리 (예: US1, US2, US3)
- 설명 내에 정확한 파일 경로를 포함하세요.

## 경로 규칙 (Path Conventions)

- **단일 프로젝트**: `src/`, `tests/` (저장소 최상단)
- **웹 앱**: `backend/src/`, `frontend/src/`
- **모바일**: `api/src/`, `ios/src/` 또는 `android/src/`
- 아래 예시에 작성된 경로는 단일 프로젝트를 가정하고 있습니다. `plan.md` 구조에 맞게 수정하세요.

## Phase 2: 기반 작업 (Blocking Prerequisites)

**목적**: 어떠한 사용자 스토리를 시작하기 전에라도 반드시 완료되어야 하는 핵심 인프라

**⚠️ 주의 (CRITICAL)**: 해당 단계(Phase)가 완료될 때까지 사용자 스토리를 진행할 수 없습니다.

기반 작업의 예시 (프로젝트에 맞게 수정 진행):

- [ ] T004 데이터베이스 스키마 및 마이그레이션 프레임워크 설정
- [ ] T005 [P] 인증/인가 (Authentication/Authorization) 프레임워크 구현
- [ ] T006 [P] API 라우팅 및 미들웨어 구조 설정
- [ ] T007 모든 스토리가 의존하는 기본 모델(Model)/엔티티(Entity) 생성
- [ ] T008 에러 예외 처리 및 로깅 인프라 구성

## 의존성 및 태스크 실행 순서 (Dependencies & Execution Order)

### 단계별 의존성 흐름 (Phase Dependencies)

- **기본 설정 (Phase 1)**: 대기할 의존성 단계가 없습니다 - 가장 먼저 즉시 시작 가능합니다.
- **기반 작업 (Phase 2)**: 기본 설정(Phase 1) 단계 완료 시 시작 - 이후의 모든 사용자 스토리에 대한 블로킹(차단) 요소입니다.
- **사용자 스토리 (Phase 3+)**: 반드시 기반 작업 단계의 선행 완료를 요망합니다.
  - 기반 작업이 완료되면 각 사용자 스토리 작업은 병렬로 진행 가능합니다 (팀에서 인력을 조율한 경우).
  - 인력이 갖춰지지 않은 경우 우선순위 순서대로 (P1 → P2 → P3) 진행합니다.
- **마무리 단계 (최종 Phase)**: 모든 사용자 스토리가 구현 완료된 이후에만 시작할 수 있습니다.

### 스토리 단위 진행 (User Story Dependencies)

- **사용자 스토리 1 (P1)**: 기반 작업 완료 후 바로 시작 가능 - 타 스토리와 의존성 연결이 없습니다.
- **사용자 스토리 2 (P2)**: 기반 작업 완료 후 시작 가능 - US1과 통합될 수 있으나 독자적으로 검증할 수 있어야 합니다.
- **사용자 스토리 3 (P3)**: 기반 작업 완료 후 시작 가능 - US1/US2와 연계되더라도 독립적 검증이 가능해야 합니다.

### 유닛 스토리 세부 단계 가이드라인

- 기능 구현 전에 테스트 코드(있는 경우)를 반드시 실패 상태(FAIL)로 먼저 작성해야 합니다.
- (서비스 계층을 작성하기 전에) 기반 모델 객체를 먼저 설계하세요.
- (컨트롤러/엔드포인트를 구현하기 전에) 서비스 로직을 완성하세요.
- 복잡한 통합 이전에 코어 모듈을 단독으로 개발하세요.
- 우선순위가 높은 스토리 구현이 완료된 뒤에야 다음 스토리로 넘어가야 합니다.

### 병렬 진행을 위한 가이드라인

- [P] 태그가 표시된 기본 설정(Setup) 작업은 병렬로 실행 가능합니다.
- Phase 2(기반 작업) 중 [P]가 붙은 태스크들을 동시 진행 가능합니다.
- 기반 작업 모듈이 완성되면 (팀 역량이 뒷받침될 때) 여러 스토리 작업을 병렬 진행할 수 있습니다.
- 단일 사용자 스토리 안에서의 [P] 태스트들을 병렬 실행 가능합니다.
- 여러 명의 팀원이 각기 다른 유닛 스토리에 대한 작업을 분담하여 병렬 진행할 수 있습니다.

## 구현 전략 (Implementation Strategy)

### 최소 기능 구현 우선 (MVP First - 오직 사용자 스토리 1)

1. Phase 1 완료: 기본 인프라 설정
2. Phase 2 완료: 기반 아키텍쳐 작업 (가장 핵심 - 다른 모든 스토리를 블로킹)
3. Phase 3 완료: 사용자 스토리 1 개발
4. **멈춤(STOP) 및 검증(VALIDATE)**: 사용자 스토리 1이 의도한 대로 독립적으로 작동하는지 확인
5. 성공적으로 확인되었다면 임시 배포/시연 진행

### 단계별 지속적 배포 (Incremental Delivery)

1. 인프라 기반 마련 완료 → 기반 시스템 구현
2. 스토리 1 적용 → 독립 테스트 → 배포/시연 (완벽한 MVP!)
3. 스토리 2 적용 → 독립 테스트 → 배포/시연
4. 스토리 3 적용 → 독립 테스트 → 배포/시연
5. 각각의 스토리는 기존 스토리를 망가뜨리지 않으면서도 시스템이 지닌 서비스 가치를 계속 업그레이드할 것입니다.

### 병렬 팀 단위 투입 (Parallel Team Strategy)

여러 개발자가 함께 작업하는 경우:

1. 하나의 팀이 뭉쳐서 기본 인프라 설정과 기반 모듈 개발을 같이 끝마칩니다.
2. 기반 모듈 개발이 완료되면 인력을 분산시킵니다:
   - A 개발자: 사용자 스토리 1
   - B 개발자: 사용자 스토리 2
   - C 개발자: 사용자 스토리 3
3. 각 스토리 작업물이 독립적으로 완성되고 무결하게 통합됩니다.

## 메모 (Notes)

- [P] 태스크 = 각기 다른 파일의 수정건이며 서로 의존성이 없음.
- [Story] 라벨 지시자는 해당 태스크를 특정 사용자 스토리와 연결(Traceability)해 줍니다.
- 각각의 사용자 스토리는 독립적인 단위로써 완성되고 검증될 수 있어야만 합니다.
- 구현 작업 전 테스트 코드가 예상대로 실패하는지를 먼저 증명해 보이세요.
- 태스크 하나 또는 관련 항목들을 묶은 논리적인 그룹이 완료될 때마다 잦은 커밋(Commit)을 수행하세요.
- 언제든지 스토리 구현 진행을 중지하고 구현이 올바른지 단위로 검사해 보세요.
- 피해야 할 항목: 애매한 목적의 태스크, 한 파일 안에서의 충돌(conflict), 다른 스토리와의 의존성을 위배하는 코드.
